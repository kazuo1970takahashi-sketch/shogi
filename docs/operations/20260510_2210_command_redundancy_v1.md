# 司令塔冗長化ルール v1.0

> ChatGPT 制限・障害時にも、AI 複数体制で安全にプロジェクト継続するための代行・縮退運転ルール。

## 1. 背景

複数 AI を組み合わせた運用体制において、**特定 AI のサービス制限・障害がプロジェクト全体を停止させない**ための冗長化が必要になった。SHOGI-TOUR A-4 の最終工程で ChatGPT・Codex の利用制限が同時発生し、Claude.ai による司令塔代行で完走に至った経験を踏まえて本ルールを明文化する。

---

## 2. 通常運転体制

| 役割 | 担当 AI | 主な責務 |
|---|---|---|
| **司令塔（最終判定）** | ChatGPT | スコープ確定、Go/No Go 判定、優先度判断、政治判断 |
| **設計レビュー** | Claude.ai | spec レビュー、設計トレードオフ評価、Codex 補完 |
| **実装** | Claude Code | コード生成・編集、テスト、PR 作成、git 操作 |
| **コードレビュー** | Codex | 実装 PR の Conditional Go / Go 判定、Must Fix 指摘 |

通常時の典型フロー：

```
Issue 起票 → Claude.ai 設計レビュー → ChatGPT 司令塔判定 (Go)
   ↓
Claude Code 実装 → Draft PR
   ↓
Codex レビュー (Conditional Go → 修正 → Go)
   ↓
ChatGPT 最終判定 → Ready 化 → マージ指示 → Claude Code マージ
```

---

## 3. 代行発動条件

以下のいずれかが成立した場合に冗長化体制へ移行：

| 条件 | 発動内容 |
|---|---|
| ChatGPT の利用制限・応答不能 | Claude.ai が司令塔を代行 |
| Codex の利用制限・応答不能 | ChatGPT or Claude.ai がコードレビューを代行 |
| Claude.ai の利用制限 | ChatGPT が設計レビューを兼務 |
| Claude Code の応答不能 | 当該プロジェクトの Claude Code セッションを再起動。代替なし |

複数同時障害時は **§5 縮退運転**を適用。

---

## 4. ChatGPT 停止時の代行体制

### 4.1 代行者
**Claude.ai** が司令塔代行を担う（ユーザー経由）。

### 4.2 代行プロトコル
- 依頼文に **「司令塔判断: Claude.ai 代行」** を明示
- 代行根拠を併記（例：「ChatGPT 制限のため」）
- 代行範囲は §6 の許可リストに限定

### 4.3 復帰時
- ChatGPT 復旧後、代行期間の判断履歴をレビュー
- ChatGPT が事後追認 or 修正指示を出す（§9）

---

## 5. Codex 停止時の縮退運転

### 5.1 通常レビューチェーン

```
Claude.ai → Claude Code → Codex → ChatGPT 最終判定
```

### 5.2 縮退レビューチェーン

```
Claude.ai → Claude Code → ChatGPT レビュー
   → Claude.ai 再確認 → ChatGPT 最終判定
```

ChatGPT がコードレビュー兼務、Claude.ai が二次確認で補完。

### 5.3 縮退時の制約
詳細は §11 参照。要点：
- **許可**：docs / test / 軽量 UI / bug fix / CI 修正
- **禁止**：Production 変更 / Security 例外判断 / DB migration / 大規模 refactor

### 5.4 復帰時
- Codex 復旧後、縮退期間にマージされた PR を Codex が事後レビュー
- A 評価未達なら follow-up PR で修正

---

## 6. Claude.ai 代行可能範囲

| カテゴリ | 代行可否 | 補足 |
|---|---|---|
| 設計レビュー（軽量〜中規模） | ✅ 可 | 通常 Claude.ai 担当のため拡張なし |
| Conditional Go の判定 | ✅ 可 | Codex 不在時のみ |
| Go / No Go 最終判定（軽量変更） | ✅ 可 | docs / dep 単発 / 軽量 UI |
| マージ可否判定（条件付き） | ✅ 可 | §7 禁止リストに該当しないこと |
| 優先度・スコープ判断 | ✅ 可 | ChatGPT 不在時 |
| 緊急 hotfix の Go 判定 | ✅ 可 | リスク評価をユーザーと事前共有 |
| ドキュメントの構造判断 | ✅ 可 | spec の改版判断含む |

---

## 7. Claude.ai 代行禁止範囲

| カテゴリ | 理由 |
|---|---|
| Production 変更（本番 DB / 本番設定） | 影響範囲がプロジェクト境界を越える |
| Security 例外判断（Scan failure 許容等） | DevSecOps 原則は司令塔本体が担う |
| DB migration の Go 判定 | rollback 困難、本番影響 |
| 大規模 refactor の Go 判定 | 影響範囲評価に司令塔本体の判断が必要 |
| 新規プロジェクト立ち上げ | スコープ・予算・体制の設計責任 |
| 体制変更（AI 役割の恒久的再定義） | ガバナンス変更は司令塔本体のみ |
| 商用契約・対外コミュニケーション | 法的・対外的責任 |

代行範囲外の判断は**ChatGPT 復旧を待つか、ユーザーに直接判断を仰ぐ**。

---

## 8. 代行履歴記録義務

### 8.1 記録媒体
- 代行が発生したプロジェクトの `docs/operations/` または `docs/specs/` 配下に記録
- 完了報告 doc に「Claude.ai 代行」の旨を明記（A-4 完了報告 §3 参照）

### 8.2 記録項目
| 項目 | 内容 |
|---|---|
| 代行発動日時 | YYYY-MM-DD HH:MM (JST) |
| 発動理由 | 例: ChatGPT 制限 |
| 代行者 | Claude.ai / ChatGPT / 他 |
| 代行範囲 | 例: dep PR の Go 判定 |
| 影響を受けた PR / Issue | 例: PR #34, Issue #36 |
| 事後レビュー予定 | ChatGPT 復旧後 / 不要 |

### 8.3 PR / Issue 上の表記
- PR 説明文に「司令塔判断: Claude.ai 代行（理由: ChatGPT 制限）」を明示
- マージコミットには反映しない（PR 説明文で十分）

---

## 9. 事後レビュー必須化

### 9.1 対象
- 縮退運転中にマージされた**全 PR**
- Claude.ai 代行で Go 判定された**全判断**

### 9.2 実施タイミング
- 各 AI 復旧後の**最初の運用ターン**で実施
- ChatGPT 復旧 → 直前の代行判断を追認 or 修正
- Codex 復旧 → 縮退期間の PR を事後レビュー

### 9.3 結果記録
- 事後レビュー結果は同 Issue / 同 spec の §13 履歴に追記
- 修正が必要なら follow-up PR を起票

---

## 10. 代行解除条件

### 10.1 自動解除
- 該当 AI が応答可能になった時点で**自動的に通常体制に復帰**
- 代行宣言の明示解除指示は不要

### 10.2 手動解除
- ユーザーまたは ChatGPT が明示的に「通常体制に戻す」と宣言
- 「Claude.ai 代行モード終了」を依頼文に明記

### 10.3 解除時の責務
- §9 の事後レビューを実施
- 代行履歴を確定（追記不可状態へ）

---

## 11. 縮退時 禁止事項

| 禁止 | 理由 |
|---|---|
| **Production 変更**（本番 DB, 本番 API キー, 本番設定） | rollback 困難、対外影響大 |
| **Security 例外判断**（Scan failure 許容等） | 司令塔本体の決裁が必要 |
| **DB migration**（schema 変更, データ移行） | 本番影響、ロールバック計画必須 |
| **大規模 refactor**（複数モジュール横断） | 影響範囲評価が代行範囲を超える |
| **依存ライブラリの major バージョン更新** | 破壊的変更のリスク |
| **新規 AI / 新規ツールの導入** | 体制変更は司令塔本体 |
| **対外公開**（release / deploy / publish） | 商用判断 |

## 11.5 縮退時 許可事項

| 許可 | 補足 |
|---|---|
| **docs**（spec, 完了報告, ops ドキュメント） | レビュー軽量で可 |
| **test** 追加・修正 | 既存実装の挙動を変えない範囲 |
| **軽量 UI**（label, 色, レイアウト微調整） | 機能変更を伴わないこと |
| **bug fix**（影響範囲が限定的なもの） | 同 PR 内で test 追加必須 |
| **CI 修正**（workflow 設定, lint 設定 等） | 本番影響なし |
| **dep の patch / minor 更新**（脆弱性対応含む） | 単発 PR、最小差分 |

## 12. 今回の A-4 で実証されたこと

| # | 事象 | 検証された設計 |
|---|---|---|
| 1 | ChatGPT 制限下で Claude.ai が司令塔代行 | §4 の代行プロトコルが機能 |
| 2 | Codex 制限下で Claude.ai が補完レビュー | §5 縮退チェーンが機能 |
| 3 | Security Scan failure → 別 PR (#34) で先行解消 → 本体 PR (#33) マージ | §11 許可（dep の patch 更新）＋ §6 不該当（Production 変更でない）で適切に進行 |
| 4 | Claude Code が「既に完了」と先回り判断 → ユーザー interrupt → 再実行 | 冗長化中こそ冪等指示を素直に再実行する原則の重要性が露呈 |
| 5 | Claude.ai 代行判断が事実関係（spec 不在）と異なる場面 → Claude Code が検証実行 → ユーザー再判断 | §10.2「代行判断の前提検証」が機能 |
| 6 | E2E failure を別 Issue (#36) として分離 | §11 許可（CI 修正は別 PR で）＋ スコープ規律が成立 |
