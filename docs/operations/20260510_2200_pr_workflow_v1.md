## 1. 基本原則

| 原則 | 内容 |
|---|---|
| **小さく分けてマージ** | 1 PR = 1 目的。spec / 実装 / dep / docs を混ぜない |
| **Draft 先行** | すべての PR は Draft で作成。Ready 化はレビュー通過後 |
| **明示マージ** | マージは別プロンプトでの明示指示後にのみ実行。Claude Code が自走でマージしない |
| **冪等指示は素直に再実行** | 「もう完了済み」と先回り判断せず指示通り実行（idempotent な操作なら害がない） |
| **Issue は議論履歴 / docs/specs は確定仕様（SSoT）** | 齟齬時は SSoT 優先 |
| **「コミット済み」と「main 反映済み」を明確分離** | Section 9 を厳守 |

---

## 2. PR ライフサイクル

```
[Phase 0] Issue 起票（議論・要件・背景）
    ↓
[Phase 1] 仕様書 PR (docs/specs/) — SSoT 確定
    ↓ レビュー → マージ
[Phase 2] 実装 PR — SSoT に従う
    ↓ Draft 作成 → Codex / Claude.ai レビュー → 修正 → Ready 化 → マージ
[Phase 3] 完了記録 PR — docs/specs/ に完了報告 + spec §13 履歴追記
    ↓ レビュー → マージ → Issue 自動 close
```

各 Phase の責務：

| Phase | 責務 | 主成果物 |
|---|---|---|
| 0 | 議論と意図の合意 | Issue（#番号） |
| 1 | 実装契約の確定 | `docs/specs/<TS>_<project>_<phase>_specs.md` |
| 2 | 契約に沿った実装 | コード + テスト |
| 3 | 完走記録と次フェーズ引き継ぎ | `docs/specs/<TS>_<project>_<phase>_completion.md` |

---

## 3. 仕様書 PR → 実装 PR → 完了記録 PR の3段階運用

### 3.1 仕様書 PR
- ブランチ命名：`docs/<project>-<phase>-spec` 等
- 内容：`docs/specs/<TS>_<project>_<phase>_specs.md` のみ
- レビュー：Claude.ai（設計）→ ChatGPT 司令塔（最終確定）
- マージ後、**main に SSoT が乗った状態を必ず作る**

### 3.2 実装 PR
- ブランチ命名：`feature/<phase>-<short-name>` 等
- 内容：コード + テスト + 必要な fixture
- **`docs/specs/` には触らない**（spec は前段で確定済み）
- レビュー：Codex（コード）→ Claude.ai 代行（補完）
- Conditional Go の Must Fix は同 PR 内で解消し再レビュー

### 3.3 完了記録 PR
- ブランチ命名：`docs/<phase>-completion` 等
- 内容：
  - `docs/specs/<TS>_<project>_<phase>_completion.md`（新規）
  - `docs/specs/<TS>_<project>_<phase>_specs.md` の §13 履歴に「完了」記録を追記
- コミットメッセージに **`closes #<issue>`** を必ず含める（Issue 自動 close）
- マージで Issue 完走が同時に成立

### 3.4 順序の厳格性
spec PR が main に未マージのまま実装 PR を立てない。**実装 PR の base は SSoT がある main を前提**にする。やむを得ず実装 PR が先行している場合は、完了記録 PR で `git cherry-pick` により spec を main へ landing する救済が許容される（A-4 の事例）。

## 4. Draft PR 運用

| 状態 | 意味 | 許可される操作 |
|---|---|---|
| Draft | 作業中・レビュー前 | push, force-push（自分のブランチに限る）, 説明追記 |
| Ready | レビュー通過済み・マージ待ち | マージのみ（追加 push は基本しない） |
| Merged | main 反映済み | 操作なし |

- **Draft → Ready 化は明示指示が必要**
- Codex / Claude.ai レビューで A-（Conditional Go）以上を取得してから Ready 化
- Ready 化と同時にマージしない（Ready 化 → 別プロンプトでマージ指示 → マージ）

---

## 5. Codex レビュー必須化

### 5.1 通常時
- すべての**実装 PR** は Codex レビューを受ける
- Codex 評価が `A` または `A-/Go` の場合のみマージ可
- `B/Conditional Go` の場合は Must Fix を同 PR 内で解消し再レビュー

### 5.2 Codex 制限時
- `docs/operations/20260510_2210_command_redundancy_v1.md`（司令塔冗長化ルール）に従う
- 縮退運転中は許可スコープを限定（Section 11 参照）

### 5.3 軽量 PR の例外
- **docs only PR / dep 単独更新 PR / CI 設定 PR** は Codex 軽量レビューで可
- ただし「軽量で可」を判断するのは Codex / Claude.ai 側であり、Claude Code は自己判断しない

## 6. Security Scan failure 対応原則

### 6.1 原則
**Security Scan が fail している PR は原則マージ禁止**。

### 6.2 例外条件
以下を**すべて満たす**場合に限り、司令塔（ChatGPT or Claude.ai 代行）の判断でマージ可：

1. failure が**当該 PR 起因ではない**（main 由来既存問題）
2. failure 内容が**当該 PR の差分と論理的に独立**
3. **別 PR で先に解消**することが困難（緊急修正など）
4. 司令塔が DevSecOps リスクを引き受ける旨を明示

### 6.3 推奨ワークフロー（A-4 で実証）
```
[1] dep 修正の単独 PR を先行マージ
    ↓
[2] 本体 PR を base 更新（gh pr update-branch）
    ↓
[3] 本体 PR の CI で Security Scan green を確認
    ↓
[4] 本体 PR をマージ
```

### 6.4 npm audit fix の最小差分原則
- `package.json` を変更しない（lockfile 一本に絞る）
- 単一脆弱性に対して 1 PR
- 依存チェーンを `npm ls <pkg>` で確認してから修正方針を決める

## 7. E2E failure 切り分け原則

### 7.1 原則
**E2E failure は本 PR 起因か既存 main 由来かを切り分ける**。

### 7.2 切り分け手順
1. main 直近 commit の CI 履歴を `gh run list --branch main --limit 5` で確認
2. main でも同じ failure があれば → **本 PR 非起因**
3. 本 PR で初めて発生したものなら → **本 PR 起因**

### 7.3 既存 failure の扱い
- 当該 PR ではマージブロックしない（Security Scan とは扱いを分ける）
- ただし**別 Issue 化してスコープ分離**する（A-4 では Issue #36 として登録）
- 別 Issue は次フェーズ着手前の Should Fix として扱う

## 8. Issue close 担当 PR 明記ルール

### 8.1 close キーワード
- **`closes #<番号>` / `fixes #<番号>` / `resolves #<番号>`** を PR or コミットメッセージに含めると GitHub が自動 close する
- `refs #<番号>` は close しない（参照のみ）

### 8.2 担当 PR の明示
- Issue を close する PR は**1 つに絞る**
- 通常は **完了記録 PR**（Phase 3）で close する
- 実装 PR で close すると spec / 完了報告が後追いになり整合性が崩れる

### 8.3 手動 close は最終手段
- PR マージで自動 close できる設計を優先
- 手動 close する場合は理由をコメントで明記

## 9. 「コミット済み」と「main 反映済み」の明確分離

### 9.1 用語の厳格化
| 用語 | 意味 |
|---|---|
| **作業ツリーで編集済み** | `git status` で `Modified:` / untracked にあがっている |
| **ステージ済み** | `git add` 後・`git diff --cached` で見える |
| **コミット済み（branch 名 X）** | branch X の HEAD に含まれる |
| **push 済み（remote/X）** | origin/X に反映 |
| **PR 化済み（#番号）** | Draft または Ready の PR が存在 |
| **main 反映済み** | `git ls-tree origin/main` でファイルが取得できる、または `origin/main` に commit が含まれる |

### 9.2 報告時の明示
- 「コミット済み」と単独で書かない
- 必ず**どのブランチか**と**main 反映済みか**を併記
- 例：「`docs/a4-completion` ブランチにコミット済み（main 未反映、PR #35 で landing 予定）」

### 9.3 検証コマンド
- main にあるか：`git ls-tree origin/main <path>`
- ある branch にあるか：`git log --all --oneline -- <path>` → ヒット commit を `git branch --contains <SHA>`

## 10. Claude.ai 代行時の確認ルール

### 10.1 代行宣言
- 司令塔代行（Claude.ai 等）が判断を下した場合、**依頼文に「司令塔判断: Claude.ai 代行」と明示**する
- 代行根拠（ChatGPT 制限等）を併記

### 10.2 検証要請
- Claude.ai 代行の判断が事実関係（リポジトリの状態、CI 結果等）に依存する場合、**Claude Code は判断前に状態確認の検証を実行**する
- 検証結果が代行判断の前提と異なる場合、停止して再判断を仰ぐ（A-4 の spec 不在発見の事例）

### 10.3 代行範囲の遵守
- 代行可能範囲（司令塔冗長化ルール §6）を超える判断は実行しない

## 11. 緊急時縮退運転

詳細は `docs/operations/20260510_2210_command_redundancy_v1.md` 参照。

要点：
- ChatGPT 制限時 → Claude.ai 司令塔代行
- Codex 制限時 → ChatGPT or Claude.ai 補完レビュー
- 縮退中は許可スコープを限定（docs / test / 軽量 UI / bug fix / CI 修正）
- Production 変更・Security 例外判断・DB migration・大規模 refactor は禁止

---

## 12. 禁止事項

| 禁止 | 理由 |
|---|---|
| **main 直接 push** | レビュー機構を回避する |
| **明示指示なきマージ** | 自走によるスコープ逸脱を防ぐ |
| **Security Scan fail のままマージ**（例外条件外） | DevSecOps 原則違反 |
| **Spec PR を経ずに実装 PR を立てる** | 実装契約の不在 |
| **1 PR に複数目的を混ぜる** | レビュー・rollback の困難化 |
| **依頼文の禁止事項を後続指示で上書き** | 明示解除指示なしに前提を変えない |
| **Issue / PR の手動 close（自動 close 可能な場合）** | 整合性の劣化 |
| **branch 削除（明示指示なし）** | 履歴・リカバリ手段の喪失 |
| **`--no-verify` での hook skip** | 安全機構の意図的迂回 |
| **`force push` to 共有ブランチ（main / develop 等）** | 履歴破壊 |

## 13. 今回の A-4 から得た教訓

| # | 教訓 | 対応 |
|---|---|---|
| 1 | spec が main に未マージのまま実装 PR を進めると、完了記録時に整合性が崩れる | §3.4 で順序を厳格化、cherry-pick による救済を許容 |
| 2 | dep の transitive 依存源を誤認すると修正方針を誤る（例: fast-uri は playwright ではなく html-validate 経由） | §6.4 で `npm ls <pkg>` 必須化 |
| 3 | E2E と Security Scan を同列に扱うとマージ判断がブレる | §6（原則禁止）と §7（既存問題は許容＋別 Issue）を分離 |
| 4 | 「コミット済み」表現が main 反映済みと混同される | §9 で用語を厳格化 |
| 5 | Codex Conditional Go 修正後の再レビュー → A-/Go の改善ループが機能 | §5 で必須化 |
| 6 | 同一指示の再送に対し「既に完了」と先回り判断するのは誤り | §1「冪等指示は素直に再実行」 |
| 7 | Issue close は完了記録 PR で `closes #` を含めるのが最も整合的 | §8 で明文化 |
| 8 | Draft → Ready → マージを別ステップに分けることで安全運用できた | §4 で明文化 |
