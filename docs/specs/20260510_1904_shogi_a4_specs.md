## 1. 品質評価しきい値

### 警告条件（A-4暫定）

以下のいずれかに該当する場合、警告を表示する。

| 指標 | しきい値 | 挙動 |
|---|---|---|
| `rematchCount` | `> 0` | 警告 |
| `avoidableWinDiffPairs` | `> 0` | 警告 |
| `maxWinDiff` | `>= 2` | 警告 |

### 表示/参考用（警告対象外）

| 指標 | 用途 |
|---|---|
| `totalWinDiff` | 参考表示のみ |
| `sameScorePairCount` | 参考表示のみ |

### 警告の挙動

- **表示のみ**
- 操作ブロックなし
- 現場運営での例外判断を妨げない

---

## 2. 理由表示フォーマット

### 採用方式

**軽量ラベル方式（並列）** を採用。文章型説明はA-5送り。

### 表示例

```
03｜片山 vs 07｜大金
[同勝数][再戦なし]

04｜佐藤 vs 08｜大野
[勝数差1][要確認][手動変更]
```

### ラベル一覧（A-4対象）

| ラベル | 表示条件 |
|---|---|
| `[同勝数]` | 両者の勝数が同じ |
| `[勝数差N]` | 勝数差がNある（N >= 1） |
| `[再戦なし]` | 過去対戦履歴なし |
| `[再戦]` | 過去対戦履歴あり |
| `[要確認]` | 警告条件のいずれかに該当 |
| `[手動変更]` | `lastModifiedBy === "manual"` |

## 4. lastModifiedBy

### スキーマ

```javascript
// state.pairings 内の各ペアに付与
{
  p1: "p1778...",
  p2: "p1778...",
  lastModifiedBy: "auto" | "manual"
}
```

### 仕様

| 場面 | 値 |
|---|---|
| `generatePairing` 生成時 | `"auto"` |
| `changePairing` 等の手動変更時 | `"manual"` |
| `normalizeState` で既存データに未定義の場合 | `"auto"` を補完 |

### 永続化

- あり（state.pairings 内）
- 後方互換：旧データは normalizeState で `"auto"` 補完

### 拡張余地

- A-5で `source: "auto" | "manual" | "edited"` 等の3値以上への拡張を検討
- 本A-4では2値のみ

## 6. 警告挙動

### 確定事項

- **警告は表示のみ**
- **禁止・ブロックしない**
- 現場運営での例外判断を許容

### 設計理由

実戦運営では「アルゴリズム的に最適でないが運営判断として必要」な変更が頻繁に発生する。これを技術的にブロックすると、現場運営が停止する重大リスクとなる。

---

## 7. 影響範囲（normalizeState 等）

### normalizeState 変更点

- `pairings[].lastModifiedBy` 未定義時の `"auto"` 補完を追加
- これ以外の変更なし

### 後方互換

- 既存データすべて読み込み可能であること
- entry_no は永続化しない（配列indexから算出）
- table_no は永続化しない（配列indexから算出）

### 影響を受ける関数

- `normalizeState`（lastModifiedBy 補完追加）
- `generatePairing`（lastModifiedBy: "auto" 付与）
- `changePairing`（lastModifiedBy: "manual" 付与、品質評価呼び出し、警告表示）
- 品質評価関数（新規追加）
- ペアリング表示・入力UI（番号表示・卓番号表示）

---

## 8. 非スコープ（A-5以降送り）

A-4では以下を実装しない。

- Supabase導入
- 複数端末同期
- BroadcastChannel等の同期実装
- 本格監査ログ
- `source: "auto" | "manual" | "edited"` の3値以上への拡張
- `reason` の本格永続化
- `table_no` の本格永続化
- `entry_no` の本格永続化
- ドラフト編集UI
- ロック状態
- QR/NFC
- 音声入力
- Blossom等の本格完全マッチング
- 対局カード廃止
- 文章型説明エンジン
- PC報告画面
- 結果確認モニター

## 9. 推奨実装順

```
1. B級R1終了状態の fixtures 作成
2. ペアリング品質評価関数の実装（純粋関数・副作用なし）
3. 品質評価関数の単体テスト
4. B級R2回帰テスト（fixturesに対して評価関数が avoidableWinDiffPairs > 0 を検出）
5. lastModifiedBy フラグ追加（normalizeState補完含む）
6. changePairing への警告組み込み（品質評価関数を呼ぶだけ）
7. 番号表示の主要画面展開（参加者一覧→組み合わせ→入力→PDF）
8. 卓番号表示（MF6-a）
9. 理由表示ラベル追加（MF4・品質評価関数の副産物として）
10. （時間があれば）勝者番号ボタン（MF6-b）
```

## 10. リスクと対策

| リスク | 対策 |
|---|---|
| normalizeState 後方互換崩壊 | entry_no / table_no を永続化しない |
| 警告がブロックとして実装される | 「表示のみ」を§6で明記 |
| 回帰テストが自動生成パスで偽陰性 | 品質評価関数のテストとして書く |
| 当日に旧バージョン使用で全無効 | A-5以降で「最新バージョン確認UIガード」検討 |
