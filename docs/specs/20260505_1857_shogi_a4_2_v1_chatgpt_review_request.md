# ChatGPT メタレビュー依頼：Phase A-4.2 設計仕様書 v1

shogi リポジトリ（kazuo1970takahashi-sketch/shogi）main ブランチの
`docs/specs/20260505_1857_shogi_design_phaseA4_2_v1.md`
を読んでメタレビューしてください。

## 背景

A-4 を main にマージし、iPhone 16 Plus で実機運用テストを実施した結果、過去参加者パネル経由で参加者を追加する際にクラス（A/B）を変更する直感的な手段がないことが判明しました。

クラスは「マスタに固定すべき属性」ではなく「大会ごとの編成判断」であるため、マスタには「主クラス」を持たせず、過去参加者パネルとサジェストリストの各行に「Aで追加」「Bで追加」両方のボタンを並べる設計（案B）を採用します。

## レビュー観点

仕様書 §9 に列挙した8点を中心に評価してください。

1. **設計判断の妥当性**
   - マスタに主クラスを持たせない方針は将棋大会運営の実態に合うか
   - 過去参加者パネル / サジェスト両方に A/B ボタンを並べるのが最適解か

2. **`addPlayerFromMaster` シグネチャ**
   - `(memberId, cls, master, state)` で過不足ないか
   - tournamentMeta は不要か（A-4 で yomiMap を渡したのとの一貫性）

3. **既存 addPlayer との重複ロジック**
   - 純粋関数化で共通化できるはずだが、A-4.2 でリファクタリングするのは適切か
   - リスクが大きいなら新規関数のみ追加で済ませるべきか

4. **`last_class` の扱い**
   - 強調表示のみで「拘束しない」設計は妥当か
   - ユーザーが意図せず last_class クラスに追加してしまうリスクの評価

5. **サジェストの動作変更**
   - 既存の「タップ→フォーム反映」を「A/B ボタンで直接追加」に置き換えるのは妥当か
   - 互換性・既存 e2e テストへの影響

6. **Stage 構成**
   - Stage 1〜5 の順序と粒度は妥当か

7. **実装禁止事項の漏れ**
   - §7 に列挙した 13 項目で、先取りリスクのある項目に漏れはないか

8. **テスト戦略**
   - 単体 12 / e2e 14 でカバレッジは十分か
   - 不足している境界条件はないか
