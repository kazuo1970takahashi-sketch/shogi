## 1. A-3 スコープ（確定）

### 1-1. 登録画面×マスタ接続（最優先）

**目的**: 受付時の幹事による手入力を常連参加者に限り大幅削減する

---

#### サジェストの動作フロー（確定）

1. 氏名入力欄に苗字を入力 → マスタ候補がインライン表示
2. 候補をタップ → 氏名・クラスがフォームに反映（即追加はしない）
3. 「追加」ボタンを押す → 参加者行に追加。支部員区分・中学生以下区分はマスタの前回値が自動で入っている
4. 必要なら行内ドロップダウンで支部員区分・中学生以下区分を変更

**ポイント**: 追加前に変更できるのは氏名・クラスのみ。支部員区分・中学生以下区分は追加後の行内セレクトで変更する。登録フォームに新規入力欄は追加しない（A案確定）。

---

#### member_id の保持（Must Fix 1 反映）

- サジェスト候補をタップした時点で `selectedMasterMemberId` を内部状態として保持する
- 「追加」ボタン押下時に、作成する player に `member_id: selectedMasterMemberId` を設定する
- 大会終了後のマスタ更新で、既存 member の `attendance_count` / `last_attended` が正しく更新される

**member_id 解除条件**:
- 候補選択後に氏名欄を手修正して、選択元 member.name と一致しなくなった場合は `selectedMasterMemberId` を解除する
- 解除された場合は通常の新規参加者として扱う（member_id なしで追加）

---

#### サジェスト候補の対象・除外条件（Must Fix 3 反映）

**対象**:
- branch master members のうち `deleted !== true` の member のみ

**除外**:
- `deleted=true` の member は候補に表示しない
- 既に当日登録済みの `member_id` は候補から除外する（二重追加防止）

**検索**:
- `name` と `yomi` の両方を対象に部分一致
- `normalizePersonName` / `normalizeYomi` 後に検索
- 1文字以上で候補表示、最大5件

**候補の表示内容**:
- 氏名・ふりがな・前回クラス・最終参加日

**UI動作**:
- 候補は入力欄の下にリスト表示
- 候補外をタップ or 入力欄 blur で閉じる
- 候補タップ = フォームへ反映のみ（即追加しない）

---

#### 引き継ぎ項目の詳細（Must Fix 2 反映）

| 項目 | 追加前の変更 | 追加後の変更 |
|------|------------|------------|
| 氏名 | ✅ フォームで変更可 | 行内編集 |
| クラス | ✅ フォームで変更可 | 行内セレクト |
| 支部員区分 | ❌（フォームに欄なし） | ✅ 行内セレクトで変更可 |
| 中学生以下区分 | ❌（フォームに欄なし） | ✅ 行内セレクトで変更可 |

---

#### クラス変更の対応

- 追加前: フォームのクラス欄で変更可能
- 追加後: 行内 UI でクラス変更できる場合は既存仕様に従う
- 組み合わせ開始後にクラス変更が必要な場合は削除→再登録

---

#### 見送り

- B案（50音タブ一覧から選ぶ）→ 20名規模には過剰
- QRコード受付 → 参加者層への依存度が高い
- タッチパネル受付 → 導入コストが高く同じ課題
- 登録フォームへの支部員区分・中学生以下区分の新規追加 → A-3 では過剰

### 1-2. F7 マスタ管理画面（編集・削除）

**F7-a 編集**:
- 編集対象: `name + yomi` のみ
- 参加履歴（attendance_count / last_attended 等）は編集不可
- 同名重複は warning のみ（自動統合・ブロックはしない）

**F7-b 削除**:
- 方式: tombstone（`deleted: true` + `deleted_at`）
- 物理削除はしない
- 削除済みはサジェスト候補・過去参加者パネルから非表示
- 復元 UI は A-4 以降

**誤削除時の救済路**:
- 正式救済路: 事前エクスポートしたバックアップを上書きインポートで戻す
- LocalStorage 直接編集は開発者向けの最終手段

---

### 1-3. F8 マスタバックアップ（エクスポート/インポート）

**F8 は branch master 形式のみを扱う。大会データの統合は既存マイグレ機能の責務。**

#### F8-a エクスポート
- 対象: `shogi_branch_master`（schema_version / updated_at / members / tombstone含む）
- エクスポート直後に「端末移行や誤削除に備えて保管してください」を案内

#### F8-b 上書きインポート
- 入力形式: branch master 形式のみ
- branch master 形式以外はエラー（大会データの場合は「マイグレ機能を使ってください」と案内）
- 確認ダイアログ: 「現在のマスタをすべて置き換えます。元に戻せません。[現在のマスタを破棄して上書き] [キャンセル]」
- tombstone を含めてバックアップ時点の状態に完全復元

#### F8-c マージインポート
- 入力形式: branch master 形式のみ
- `member.id` を基準に統合（name+yomiの一致だけで自動統合しない）
- **tombstone**: `deleted=true` を自動で `deleted=false` に復元しない
- **同一id衝突ルール（既存側優先）**:
  - name / yomi は既存側を維持
  - tournament_ids は union
  - last_attended は新しい日付を採用
  - first_attended は古い日付を採用
  - attendance_count は tournament_ids の件数または大きい方
  - deleted は安全側（true を維持）

## 4. 実装禁止事項

1. schema_version を変更しない
2. F8 で大会データ（players を含む tournament 形式）を処理しない
3. `updateBranchMasterFromTournament` を F8 のロジックとして使わない
4. マージインポートで tombstone を自動復元しない
5. name + yomi の一致だけで member を自動統合しない
6. 既存のマイグレ機能の責務を変えない
7. 参加履歴を F7 編集画面で編集可能にしない
8. 復元 UI を A-3 に実装しない
9. A-4 送り項目を先取り実装しない
10. build/bind 分離パターンを壊さない
11. 登録フォームに支部員区分・中学生以下区分の新規入力欄を追加しない
12. サジェスト候補選択後に氏名が変更された場合、元の member_id を保持し続けない
13. `deleted=true` の member をサジェスト候補に表示しない
14. 既に当日登録済みの member_id を二重追加しない
15. 候補タップだけで即追加しない（フォームへの反映のみ）
