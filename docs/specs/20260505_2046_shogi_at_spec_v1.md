## 4. 設計方針

### 4.1 偽陽性を構造的に排除する 4 原則

| # | 原則 | 実装手段 |
|---|------|---------|
| 1 | **実 DOM クリックを強制** | `force: true` 禁止、`page.click()` の前に hit-test を実行 |
| 2 | **クリック前のクリック可能性を検証** | カスタム expect `expectClickable(locator)` を全 click 前に挿入 |
| 3 | **クリック後の DOM 変化を検証** | クリック前後で DOM スナップショットを比較、想定変化を厳密にアサート |
| 4 | **CSS 計算値も検証** | `pointer-events`/`opacity`/`visibility`/`display`/z-index/bounding rect を必ずチェック |

### 4.2 `expectClickable` ヘルパ仕様

クリック前に以下をすべて満たすことを検証：

```javascript
async function expectClickable(locator) {
  // 1. 要素が存在する
  await expect(locator).toBeAttached();

  // 2. 表示されている（display !== 'none', visibility !== 'hidden'）
  await expect(locator).toBeVisible();

  // 3. 無効化されていない
  await expect(locator).toBeEnabled();

  // 4. 計算スタイルの厳密検証
  const style = await locator.evaluate(el => {
    const cs = getComputedStyle(el);
    return {
      pointerEvents: cs.pointerEvents,
      opacity: parseFloat(cs.opacity),
      visibility: cs.visibility,
      display: cs.display
    };
  });
  expect(style.pointerEvents).not.toBe('none');
  expect(style.opacity).toBeGreaterThan(0);

  // 5. ヒットテスト：実際のクリック座標で elementFromPoint した結果が
  //    自分自身または自分の子孫であること（オーバーレイ検出）
  const isHit = await locator.evaluate(el => {
    const rect = el.getBoundingClientRect();
    const cx = rect.left + rect.width / 2;
    const cy = rect.top + rect.height / 2;
    const top = document.elementFromPoint(cx, cy);
    return top === el || el.contains(top);
  });
  expect(isHit, 'クリック座標が他要素にブロックされている').toBe(true);
}
```

### 4.3 テストデータ生成器の設計

**100 パターンを決定論的に生成**。seed から再現可能、CI でも同じデータが得られる。

#### 8 軸の組合せ

| 軸 | バリエーション | 件数 |
|---|---|---|
| ふりがな先頭文字 | あ/か/さ/た/な/は/ま/や/ら/わ/他（11） | 11 |
| 支部員区分 | member / other（2） | 2 |
| 学年区分 | ippan / chu（2） | 2 |
| 参加回数 | 0 / 1 / 3 / 5 / 10 / 20（6） | 6 |
| 最終参加日 | 今日 / 1ヶ月以内 / 3ヶ月以内 / 半年 / 1年 / 2年（6） | 6 |
| 前回クラス | A / B / null（3） | 3 |
| 削除状態 | 通常 / tombstone / 復元済（3） | 3 |
| 名前/ふりがな特殊性 | 通常 / 空白あり / 異体字 / カタカナ / 拗音 / 濁音 / 長音 / 空（8） | 8 |

全直交：11 × 2 × 2 × 6 × 6 × 3 × 3 × 8 = **228,096 通り**

ここから seed を使った決定論的サンプリングで 100 件を抽出。

#### 必須カバレッジ（サンプリングで必ず含める）

| # | 必須ケース |
|---|----------|
| 1 | 五十音 11 タブすべてに最低 5 件ずつ |
| 2 | クイックフィルタ「前回参加」「3ヶ月以内」「常連 (5回以上)」「ふりがな未入力」すべて 5 件以上発火 |
| 3 | tombstone 状態のメンバー 10 件以上 |
| 4 | 復元済み（deleted: false, deleted_at: 過去）3 件 |
| 5 | 異体字（髙・吉・斎/齋/齊・渡邉/渡邊）含む 5 件 |
| 6 | 名前空白パターン（"山田 太郎" / "山田太郎"）両方 |
| 7 | 中学生（grade=chu）10 件 |

#### 出力フォーマット

`test/fixtures/master_100_patterns.json`（JSON）+ 生成器スクリプト `test/fixtures/generate_master.js`

### 4.4 モンキーテストランナーの設計

#### 操作種別

| 種別 | 内容 |
|------|------|
| クリック | 全ボタン・全タブ・全リンク |
| 入力 | テキスト入力欄にランダム文字列・特殊文字 |
| 選択 | セレクトボックス・チェックボックス |
| スクロール | 上下・左右 |
| 画面サイズ変更 | 320 / 375 / 430 / 768 / 1024 / 1280 / 1920 |
| タッチ | mobile emulation での tap/swipe |

#### 検証項目（毎ステップ）

1. JS エラーが発生していない（`page.on('pageerror')` で監視）
2. console.error が出ていない
3. DOM が壊れていない（特定の必須要素が存在し続ける）
4. localStorage が破損していない（パース可能・schema_version 維持）

#### 実行モード

| モード | 用途 | seed |
|--------|------|------|
| 決定論モード | CI で再現可能 | 固定 seed |
| 探索モード | ローカルで網羅探索 | ランダム seed、失敗時 seed を保存 |

## 7. 実装方針・原則

- **shogi_v4.html 単一ファイル原則維持**（テストコードのみ追加）
- **A-4.2 のロジックには触らない**（A-T では検出のみ、修正は A-4.2.1）
- **playwright を継続採用**（基盤切替はしない）
- **外部ライブラリ追加最小限**：必要なのは playwright の追加機能のみ。fixture 生成は標準 JS で。
- **CI ランタイムを 5 分以内に抑える**（モンキーテストはサンプル数を調整）
- **既存テスト名・ファイル構造は維持**（互換性保持、ただし内部実装は force click 排除に書き換え）

---

## 8. 禁止事項・スコープ外

1. shogi_v4.html のロジック層変更
2. localStorage 構造の変更（schema v1 維持）
3. テストフレームワーク切替（jest → vitest 等の移行は禁止）
4. Docker / コンテナ導入
5. 外部 SaaS 依存（visual regression を Percy 等に外出ししない、ローカル比較のみ）
6. テストの並列度を上げて結果を不安定にする変更
7. A-4.2 のバグ修正そのもの（検出は A-T、修正は A-4.2.1）

## 9. リスク・注意事項

| # | リスク | 対策 |
|---|--------|------|
| 1 | 既存 272 件のうち多数が偽陽性で書き直しが必要 | Stage 1 で実態把握、書き直しは Stage 2 で段階的に |

| 4 | テスト実装中に仕様書側の不整合発見 | 各 Stage 完了時にロードマップ・本仕様書を更新 |
| 5 | CI 時間超過で開発体験が悪化 | モンキーテスト件数を CI で 100 ステップ × 3 seed に絞る、Nightly で 1000 × 10 |
| 6 | A-4.2 バグ以外の潜在バグも一気に表面化 | A-T では「検出」のみ、修正は別フェーズに切る判断を維持 |
