### 1.2 schema_version / migration 関数の有無

- **`schema_version` あり**: `BRANCH_MASTER_SCHEMA_VERSION = 1`、export / save / load 全てで保持
- **migration 関数なし**: `normalizeBranchMaster` L457-459 で **schema_version 不一致なら `createEmptyBranchMaster()` を返す = データ破壊** という強い分岐
  ```javascript
  if(schema_version!==BRANCH_MASTER_SCHEMA_VERSION)return createEmptyBranchMaster();
  ```
- **結論(§8.1 判断)**: `BRANCH_MASTER_SCHEMA_VERSION` を 2 に bump すると、既存ユーザーの localStorage(v=1)が**ロード時に空マスタへ置換 = マスタ全消失 P0**。**bump 不可**。仕様書 §6 Out of Scope にも「schema_version 新設(§8.1 で「無し」判定時)は含めない」とあり整合。
  - 採用方針: **schema_version は 1 のまま維持**、`normalizeBranchMaster` の members ループで `city` を読み取り正規化補完(§2.1 ルール適用)→ 旧データ + 新データ両方を継ぎ目なくサポート。

### 1.3 F7 編集モーダルの実装箇所と既存入力欄の構造

`buildMasterEditModalHtml` L1731-1757 の構造:

```
氏名 input(L1731-1732)

─── 区分 fieldset 群 ───

```

**city 追加位置の判断**: 「ふりがな」の直後(L1734 の直後)に「市町村」input を追加。理由は「個人属性 → 区分 → 履歴」の論理順序を維持できるため。

### 1.4 backup/restore の JSON 形式

- export(L754-762)は `members` 配列をそのまま JSON.stringify するため、**city フィールドを member に含めれば自動的に export される**
- normalizeBranchMaster の member 再構築(L491-505)に **city を加える限り**、import 時も保持される(L649 / L668 経由)
- `applyMergeImport` L731「name/yomi/member/grade/last_class/note は既存維持」のコメント。**city も既存維持にすべきか**?
  - 採用方針: **既存維持(merge 時は city を上書きしない)**。理由: 仕様書 §2.1「手入力・JSON import・旧データ読込補完すべてで同一ルール」と矛盾しない(import の値は normalize されるが、既存の値があればそれを優先する merge ポリシーは F8 の元方針)。仕様書では明示されていないので**実装側判断**として記録、Codex review で確認依頼。

### 1.5 既存 e2e 594 の対象範囲(city 追加で影響を受けるテストの有無)
