### 0.2 ユーザーストーリー（v1.2.5 §2.2 L1 必須項目反映）

A-T は内部テスト基盤強化フェーズであり、エンドユーザー（沼津支部の月例参加者）に対する直接的な機能追加ではない。本フェーズの「ユーザー」は **A-T 基盤の上でテスト・レビュー・実装を行う開発者・AI 群および髙橋さん**である。以下 5 本のユーザーストーリーを必須記載とする：

#### US-1（Claude Code 視点）：実装担当として、primary assertion を「書き忘れ得ない」状態で新規テストを書きたい

> 実装担当の AI として、新規 click テストを書く時、`clickAndExpectChange` ヘルパを呼ぶだけで「primary semantic assertion を 1 つ以上宣言しないと型・実行時エラーで進めない」状態にしたい。
> 通知表示のみを成功条件にしてしまう A-4.2 型のテストを「書きたくても書けない」構造にすることで、注意力に依存しない品質を実現するため。

#### US-2（Codex 視点）：独立レビュアーとして、primary assertion 不在を機械的に検出したい

> 独立コードレビュアー（Codex）として、PR の e2e diff を読む時、`force: true` の使用 / `clickAndExpectChange` 不使用の click / 通知のみの成功条件 を機械的に grep + パースで検出し、レビュー結果に「primary assertion 不在: N 件」と機械検証可能な YAML 形式で出力したい。
> A-4.2 では Codex が B+ → 修正後 Yes と判定したが偽陽性を見抜けなかった。これを次回構造的に検出できるレビュー基盤にするため。

#### US-3（Claude.ai 視点）：設計担当として、Stage 完了時に偽陽性ゼロをワンショットで確認したい

> 設計・進行管理の AI（Claude.ai）として、Stage 完了報告を受け取った時、`force: true` 残存数 / `clickAndExpectChange` 未置換数 / primary assertion 宣言済率 / 既存 272 件の再分類進捗 を 1 つの dashboard で確認し、即座に「次 Stage に進めるか」を判定したい。
> 4 者協調体制（§10）で Stage 進行のボトルネックにならず、判断スピードを保つため。

#### US-4（髙橋さん視点）：プロジェクトオーナーとして、実機運用試験を経ずに実機リグレッションを防ぎたい

> プロジェクトオーナー（髙橋さん）として、A-T 完了後は **通常操作で再現できる** A-4.2 型の「e2e は緑だが実機で動かない」リグレッションが PR ごとに自動検出され、**実機運用試験への依存を最小化**したい。
> v1.2.5 §13「実装以降の人間ゼロ介在フロー」で段階 1 移行を可能にし、AI 開発の 10 倍速を阻害する人間レビュー律速を排除するため。端末固有・ブラウザ固有・タッチ挙動固有の問題は実機試験で別途確認する。

#### US-5（将来の AI / 人間視点）：A-T 基盤の利用者として、新規 click 操作を安全に追加できる土台が欲しい

> A-7（履歴閲覧統合）/ A-8 / A-9 等で新機能を追加する将来の AI または人間として、`expectClickable` + `clickAndExpectChange` + §4.3.7 の primary assertion カタログを土台に、新規 click 操作を「カタログに 1 行追加 + テストを 1 件書く」だけで安全に導入したい。

## 4. 設計方針

### 4.1 偽陽性を構造的に排除する 4 原則

| # | 原則 | 実装手段 |
|---|------|---------|
| 1 | **実 DOM クリックを強制** | `force: true` 禁止、`page.click()` の前に hit-test を実行 |
| 2 | **クリック前のクリック可能性を検証** | カスタム expect `expectClickable(locator)` を全 click 前に挿入 |
| 3 | **クリック後の DOM 変化を検証** | `clickAndExpectChange` ヘルパで primary semantic assertion を必須化（§4.3） |
| 4 | **CSS 計算値も検証** | `pointer-events`/`opacity`/`visibility`/`display`/z-index/bounding rect を必ずチェック |

### 4.2 `expectClickable` ヘルパ仕様（v1.2 で 7 段階検証に強化、v1.2.5 §2.2 L4 整合）

クリック前に以下をすべて満たすことを検証：

```javascript
async function expectClickable(locator) {
  // 1. 物理的存在: 要素が DOM に存在し、可視で、有効化されている
  await expect(locator).toBeAttached();
  await expect(locator).toBeVisible();
  await expect(locator).toBeEnabled();

  // 2. scrollIntoViewIfNeeded で viewport 内に収める
  await locator.scrollIntoViewIfNeeded();

  // 3. 矩形検証: rect.width/height > 0、viewport 内
  const rect = await locator.boundingBox();
  expect(rect, '要素が描画されていない').not.toBeNull();
  expect(rect.width).toBeGreaterThan(0);
  expect(rect.height).toBeGreaterThan(0);

  // 4. CSS 計算値検証
  const style = await locator.evaluate(el => {
    const cs = getComputedStyle(el);
    return {
      pointerEvents: cs.pointerEvents,
      opacity: parseFloat(cs.opacity),
      visibility: cs.visibility,
      display: cs.display,
    };
  });
  expect(style.pointerEvents).not.toBe('none');
  expect(style.opacity).toBeGreaterThanOrEqual(0.5);
  expect(style.visibility).not.toBe('hidden');
  expect(style.display).not.toBe('none');

  // 5. ancestor chain 検証:
  //    display !== 'none' / visibility !== 'hidden' / inert 属性なし /
  //    aria-disabled !== 'true' / 祖先の disabled fieldset なし
  const ancestorBlocked = await locator.evaluate(el => {
    let cur = el.parentElement;
    while (cur) {
      const cs = getComputedStyle(cur);
      if (cs.display === 'none') return 'ancestor display:none';
      if (cs.visibility === 'hidden') return 'ancestor visibility:hidden';
      if (cur.hasAttribute('inert')) return 'ancestor inert';
      if (cur.getAttribute('aria-disabled') === 'true') return 'ancestor aria-disabled';
      if (cur.tagName === 'FIELDSET' && cur.disabled) return 'ancestor disabled fieldset';
      cur = cur.parentElement;
    }
    return null;
  });
  expect(ancestorBlocked, `祖先要素にブロック要因: ${ancestorBlocked}`).toBeNull();

  // 6. ヒットテスト（v1.2 で 5 点に拡張）:
  //    elementFromPoint で以下 5 点すべてが自身または子孫であることを検証
  //    これにより z-index・absolute オーバーレイ等の踏み抜きを構造的に検出
  const hitResult = await locator.evaluate(el => {
    const r = el.getBoundingClientRect();
    const points = [
      [r.left + r.width / 2, r.top + r.height / 2],   // 中央
      [r.left + 1, r.top + 1],                        // 左上内側
      [r.right - 1, r.top + 1],                       // 右上内側
      [r.left + 1, r.bottom - 1],                     // 左下内側
      [r.right - 1, r.bottom - 1],                    // 右下内側
    ];
    return points.map(([x, y]) => {
      const top = document.elementFromPoint(x, y);
      return top === el || (top && el.contains(top));
    });
  });
  expect(hitResult.every(Boolean), `5 点 hit-test の一部が他要素にブロックされている: ${JSON.stringify(hitResult)}`).toBe(true);

  // 7. CSS 由来の踏み抜き要因（pointer-events / visibility / z-index）も検証対象に含める
  //    → 上記 4・5・6 ですでに網羅
}
```

**重要な禁止事項**（v1.2 で明文化、v1.2.5 §2.2 L4 整合）：
- **L4 ヘルパ内では `force: true` の使用を禁止**
- これにより playwright が `pointer-events: none` を踏み抜くバグ（A-4.2 の根本原因）を構造的に防止
- リポジトリ全体で `force: true` の使用箇所をゼロにする（Stage 2 完了基準）

### 4.3 `clickAndExpectChange` ヘルパ仕様（v1.2 で新設、v1.2.5 §2.2 L4 整合）

#### 4.3.1 役割

`expectClickable` が「クリック前の物理的可能性」を検証するのに対し、`clickAndExpectChange` は「クリック後の意味的成功条件」を検証する。
A-4.2 リグレッションは「e2e は緑だが実機で動かない」という構造的欠陥に由来し、これを防ぐには **クリックが業務状態を変えたかを assert する** 必要がある。

#### 4.3.2 仕様

```javascript
async function clickAndExpectChange(locator, expectedChange) {
  // 1. クリック前検証
  await expectClickable(locator);

  // 2. クリック前 state スナップショット取得
  const beforeState = await getStateSnapshot(); // テスト側で定義

  // 3. クリック実行（force: true 禁止）
  await locator.click();

  // 4. クリック後 state スナップショット取得
  const afterState = await getStateSnapshot();

  // 5. expectedChange を検証（primary semantic assertion を含む）
  await expectedChange(beforeState, afterState);
}
```

`getStateSnapshot()` の実装はテスト側で定義する。shogi_v4 では `window.state` および `localStorage` の主要キーを取得する形を推奨。

#### 4.3.3 primary semantic assertion 必須化

`expectedChange` には以下の **primary semantic assertion を 1 つ以上必須**で含める：

| 種類 | 内容 | 適用例 |
|------|------|--------|
| **状態変化** | `state` / `store` / `global vars` の差分 | `state.players[lastIndex].cls === 'A'` |
| **DOM 変化** | 要素追加・削除・属性変化（DOM が業務状態を反映する場合） | 登録者一覧の `<li>` 行追加 |
| **永続化変化** | localStorage / IndexedDB / cookie の差分 | 保存ボタン: `JSON.parse(localStorage.getItem('players'))` 一致 |
| **URL 変化** | pathname / hash / query の変化 | 画面遷移系操作 |

#### 4.3.4 補助 assertion（任意・追加）

primary に加えて、以下を補助として含めることができる：

- 通知表示（showMsg / toast / alert）
- アニメーション完了
- その他の二次的変化

#### 4.3.5 重要な禁止事項（A-4.2 型悪用防止）

- **UI 重要操作（v1.2.5 §1.4）では、通知表示のみ を成功条件とすることを禁止**
- 例：保存ボタンで `expect(showMsg).toBeVisible()` だけを成功条件にし、実データが保存されていない、は不可
- primary assertion を必ず 1 つ以上含めること

#### 4.3.6 画面遷移系操作の例外（v1.2.5 v2.2.1 整合）

- 画面遷移・タブ遷移が**業務目的そのもの**の場合（例：「ホーム画面に戻る」「設定画面を開く」）、URL 変化を primary として単独で許容する
- ただし**遷移先の主要 DOM 表示確認**を補助 assertion として推奨

#### 4.3.7 shogi 固有の primary assertion カタログ

| 操作 | primary semantic assertion |
|------|---------------------------|
| クラスボタン A | `expect(state.players[lastIndex].cls).toBe('A')` |
| クラスボタン B | `expect(state.players[lastIndex].cls).toBe('B')` |
| 削除ボタン | `expect(state.players.length).toBe(beforeLength - 1)` |
| 保存ボタン | `expect(JSON.parse(localStorage.getItem('players'))).toEqual(state.players)` |
| 過去参加者から追加 | `expect(state.players.length).toBe(beforeLength + 1)` + member_id 一致 |
| マスタ編集削除 | `expect(state.master.member.find(m => m.id === target).deleted).toBe(true)` |
| エクスポート | `expect(JSON.parse(clipboardText)).toMatchSchema(exportSchema)` |
| インポート（マージ） | `expect(state.master.member.length).toBe(beforeLength + newCount)` |

このカタログは Stage 2 着手時に refactor / 拡張する。

#### 4.3.8 新規 click 操作の追加ルール（v1.3 新設・ChatGPT Should Fix #1 反映）

A-7 / A-8 / A-9 等で新規 click 操作を追加する場合、以下を必須とする：

1. **操作名を §4.3.7 primary assertion カタログに追加する**（カタログにない操作のテストを書くことを禁止）
2. **操作の業務目的を 1 文で記載する**（「ユーザーは何を達成したいのか」）
3. **primary semantic assertion を state / DOM / 永続化 / URL のいずれかで最低 1 つ定義する**
4. **UI 重要操作（v1.2.5 §1.4）の場合、通知表示のみを成功条件にしない**
5. **既存カテゴリに当てはまらない場合は、Stage 着手前に ChatGPT / Codex レビューへ回す**

判定責任：

- **新規操作の業務目的整理**：Claude.ai（仕様書段階）
- **primary assertion の妥当性確認**：ChatGPT メタレビュー
- **カタログ反映と実装**：Claude Code（Stage 着手時）
- **PR レビューでの抜け検出**：Codex（v1.2.5 §13.4 YAML フォーマットで機械検証）

## 7. 実装方針・原則

- **shogi_v4.html 単一ファイル原則維持**（テストコードのみ追加）
- **A-4.2 のロジックには触らない**（A-T では検出のみ、修正は A-4.2.1）
- **playwright を継続採用**（基盤切替はしない）
- **外部ライブラリ追加最小限**：必要なのは playwright の追加機能のみ。fixture 生成は標準 JS で。
- **CI ランタイムを 5 分以内に抑える**（モンキーテストはサンプル数を調整）
- **既存テスト名・ファイル構造は維持**(互換性保持、ただし内部実装は force click 排除に書き換え)

**CI ランタイム 5 分以内の位置づけ**（v1.3 で明確化・ChatGPT Should Fix #5 反映）：

- 「CI ランタイム 5 分以内」は shogi A-T における **PR CI の努力目標** であり、**v1.2.5 全体の SLA ではない**
- v1.2.5 §6.4 では CI ランタイム上限は SLA としては未定、本格対処は v2.0
- 5 分超過時は**即 merge blocker としない**。テスト分割、Nightly 移行、PR CI 軽量化を検討する
- Release / Nightly では 5 分超過を許容する
- 努力目標として 5 分を目指すが、達成できなくても Stage 完了を阻害しない
