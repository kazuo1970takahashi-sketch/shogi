## 1. 概要

### 1.1 Phase A-3 の位置づけ

Phase A-2 でふりがな対応の3点セット（F4/F5b/F6）+ Codex Minor 2件が完成・main マージ済み。
Phase A-3 では、運用試験を待たずに **マスタ運用の補完機能** を実装する。

理由：
- A-1/A-2 で蓄積したマスタを **編集・削除** できないと、誤入力が永続的に残る
- 端末故障・買替時の **エクスポート/インポート** 機構が必要
- マスタが大きくなったとき **クイックフィルタ** で常連を素早く呼び出したい

### 1.2 含む機能（3項目）

| ID | 機能 | 規模 | 出典 |
|---|---|---|---|
| **F7-a/b** | マスタ編集（編集 + tombstone 削除）| 大 | A-2 v6 §7 で予約 |
| **F8** | マスタのエクスポート/インポート（戦略 Z）| 中 | A-2 v6 §7 で予約 |
| **クイックフィルタ** | 前回参加者・3ヶ月以内・常連（排他選択）| 小 | A-2 v6 §7 で予約 |

### 1.3 含めない機能（A-4 以降）

- F7-c マスタタブ内検索（過去参加者パネル側に既に検索があるため冗長）
- F7-d 削除済みの復元（tombstone リカバリ、使用頻度低）
- マイグレーション対話 UI（プレビュー、統合/別人選択、A-1 の「複数候補は新規扱い」運用が確立）
- 保留 participant の手動紐付け UI（F7-a 編集で member 統合可能）
- 利用目的表示の詳細化（A-2 で「今後提供予定」に更新済）

これらの優先順位は **A-3 リリース後の運用試験フィードバック** で決定する。

## 2. データモデル変更（最小）

### 2.1 支部マスタ schema_version

A-2 v6 から **変更なし**（schema_version: 1 のまま）。
`deleted: boolean` / `deleted_at: string` フィールドは A-1 v5 §3.1 で予約済み。A-3 で活用を開始する。

```json
{
  "schema_version": 1,
  "members": [
    {
      "id": "m_550e8400e29b",
      "name": "佐藤太郎",
      "yomi": "さとうたろう",
      "deleted": false,        ← A-3 で tombstone 化に活用
      "deleted_at": null,      ← A-3 で削除日時記録に活用
      ...
    }
  ]
}
```

### 2.2 後方互換性

- A-1/A-2 で構築したマスタの `deleted: false` をそのまま読める
- A-3 で削除された member（`deleted: true`）を A-1/A-2 が読んでも壊れない
  - A-1/A-2 では `members.filter(m => !m.deleted)` のロジックが既に入っている
  - A-3 で削除した member は A-1/A-2 でも非表示になる（自然な互換性）

### 2.3 エクスポート/インポートのデータ形式

エクスポート/インポートする JSON は LocalStorage 保存形式と同一：

```json
{
  "schema_version": 1,
  "updated_at": "2026-05-05T09:41:00.000Z",
  "members": [...]
}
```

ファイルには tombstone（`deleted: true`）も含める。理由：
- 「削除済みの履歴」を別端末でも引き継ぎたい
- 同一 member の重複削除登録を防げる

## 3. 機能仕様

### 3.1 F7-a / F7-b: マスタ編集 + 削除

#### 3.1.1 起動

「マスタ」タブの簡易一覧の各行に「編集」ボタンを追加。タップで編集モーダル表示。

```
沼津支部 参加者マスタ
登録: 87名

[簡易一覧]
氏名         最終参加日   回数  操作
佐藤太郎     2026-04-15   12    [編集]
鈴木一郎     2026-04-15   8     [編集]
田中三郎     2026-03-15   5     [編集]
...
```

#### 3.1.2 編集モーダル仕様

```
┌─ 参加者の編集 ───────────────────────────┐
│ 氏名     [佐藤太郎              ]         │
│ ふりがな [さとうたろう          ]         │
│                                          │
│ 最終参加日: 2026-04-15                    │
│ 参加回数: 12回                            │
│ 初回参加: 2026-01-15                      │
│                                          │
│ [保存]  [削除]  [キャンセル]               │
└──────────────────────────────────────────┘
```

**編集可能項目**：
- 氏名（name）
- ふりがな（yomi）

**読み取り専用項目**：
- last_attended（最終参加日）
- attendance_count（参加回数）
- first_attended（初回参加日）
- id（member_id）
- tournament_ids（参加大会リスト）

理由：参加履歴は事実なので手動編集すべきでない。誤って削除する場合は §3.1.3 削除機能を使う。

#### 3.1.3 削除動作（tombstone）

「削除」ボタンを押した時：

```
┌─ 削除の確認 ──────────────────────────────┐
│ 「佐藤太郎」を支部マスタから削除します。  │
│                                          │
│ ・参加履歴（12回）も一緒に消えます        │
│ ・過去参加者パネルから消えます            │
│ ・大会開催で再度同名が登録された場合は    │
│   新規 member として追加されます          │
│                                          │
│ [削除を実行]  [キャンセル]                 │
└──────────────────────────────────────────┘
```

確認後の動作：
- `member.deleted = true`
- `member.deleted_at = new Date().toISOString()`
- saveBranchMaster で保存
- マスタタブの簡易一覧から消える
- 過去参加者パネルからも消える（既存の `filter(m => !m.deleted)` が効く）

**注意**：
- member は **物理削除しない**（tombstone のみ）
- 削除された member は A-3 では UI 上は復元できない（F7-d は A-4 送り）
- LocalStorage を直接いじれば復元可能（運用想定外の救済路）

#### 3.1.4 編集動作

「保存」ボタンを押した時：

- 入力された name と yomi を `normalizePersonName` / `normalizeYomi` で正規化
- name が空文字 → エラー「氏名は必須です」
- yomi は空文字 OK（既存仕様通り）
- saveBranchMaster で保存
- マスタタブの簡易一覧と過去参加者パネルが再描画される

**重複チェック**：
- 編集後の name + yomi が他の member（自分以外）と重複した場合の処理
- → 警告のみ（運用上、同名別人もあり得るため強制統合しない）

```
showMsg('同名の member が他にも存在します。マスタを確認してください。', 'warn')
```

### 3.2 F8: エクスポート/インポート

#### 3.2.1 起動

マスタタブに2つのボタンを追加：

```
沼津支部 参加者マスタ
登録: 87名

[📥 過去大会を支部マスタに統合]  ← A-1 既存
[📤 マスタをエクスポート]        ← A-3 新規
[📂 マスタをインポート]          ← A-3 新規
```

#### 3.2.2 エクスポート動作

「📤 マスタをエクスポート」をタップ：

- 現在のマスタ（tombstone 含む）を JSON ファイルとしてダウンロード
- ファイル名：`shogi_branch_master_YYYY-MM-DD.json`（例：`shogi_branch_master_2026-05-05.json`）
- フォーマット：LocalStorage と同形式（schema_version: 1）

スマホ対応：
- ダウンロードは Web Share API or `<a download>` 属性
- iOS Safari ではファイル保存先を選択

#### 3.2.3 インポート動作

「📂 マスタをインポート」をタップ：

```
┌─ マスタをインポート ──────────────────────┐
│ JSONファイルを選択:                       │
│ [ファイル選択]                            │
│                                          │
│ インポート方式:                            │
│ ○ このファイルで上書き（既存マスタを破棄） │
│ ○ 既存マスタに追加（マージ）              │
│                                          │
│ [実行]  [キャンセル]                      │
└──────────────────────────────────────────┘
```

**戦略 Z（両方サポート）**：
- ラジオボタンで「上書き」「マージ」を選択
- 「上書き」選択時はさらに確認ダイアログ「現在のマスタを破棄します。本当によろしいですか？」

**上書きモード**：
- 既存マスタを破棄
- インポートしたファイルで丸ごと置換
- 端末移行用（古い端末からエクスポート → 新端末でインポート）

**マージモード**：
- 既存マスタにインポートを追加
- ロジックはマイグレウィザードと同じ（`updateBranchMasterFromTournament` ベース）
- 同名重複は新規扱い（A-1 と同じ方針）
- バックアップ統合用

**結果サマリ表示**：

```
インポート完了
読込：87名
新規追加：12名
既存統合：5件
スキップ：2件（複数候補）
```

#### 3.2.4 不正データの検証

インポートされた JSON の検証：

- `schema_version` が 1 でない → エラー「未対応のフォーマットです」
- `members` が配列でない → エラー「ファイルが破損しています」
- 個別 member のフィールド不足 → 補完または除外

`normalizeBranchMaster` を再利用して破損データを安全に処理。

### 3.3 クイックフィルタ

#### 3.3.1 配置

過去参加者パネルの50音タブの下：

```
[🔍 検索: ...]
[全][あ][か][さ][た][な][は][ま][や][ら][わ][他]
[最終参加日 直近] [3ヶ月以内] [常連 (5回以上)]   ← A-3 新規
```

#### 3.3.2 フィルタ定義

| フィルタ | 定義 | データ判定 |
|---|---|---|
| **最終参加日 直近** | 全 member の last_attended 最大日と一致 | 動的計算 |
| **3ヶ月以内** | 今日（todayYmd）から3ヶ月（90日）以内に last_attended | 日付計算 |
| **常連 (5回以上)** | attendance_count >= 5 | 単純比較 |

「前回参加者」の定義について：
- 「直近の大会」とは shogi_v4.html の現在の大会ではなく、**マスタの member の last_attended の最大値** で判定
- これにより「マスタを開いただけで前回参加者がわかる」UX

#### 3.3.3 排他選択 + 解除

- 3つのフィルタは **ラジオ的な排他選択**
- 既選択のフィルタを再タップ → 解除（フィルタなし状態に戻る）
- 50音タブと **AND 条件**（タブで絞った中でフィルタ）
- 検索ボックスとも AND

例：
- 「常連」フィルタ + 「さ」タブ + 検索「さ」 → 5回以上参加 ∩ さ行 ∩ 「さ」を含む name/yomi
- 結果：例えば「佐藤太郎（attendance:12, yomi:さとうたろう）」

#### 3.3.4 ボタン UI

選択中：青系の背景色（既存の active class と同じ）
非選択：白背景

「常連」の閾値（5回）は **定数でハードコード**（A-3 では設定変更 UI を作らない、YAGNI）

#### 3.3.5 クイックフィルタの状態保持

- 過去参加者パネルを再描画してもフィルタ選択は保持（ppYomiRow と同じスコープ）
- 大会リセット時にもフィルタ状態は保持（マスタはリセット対象外と整合）
- 別タブに切り替えて戻ってきても保持

## 5. テスト計画

### 5.1 既存テスト維持（最重要）

A-2 v6 時点の全テスト（既存 50 + 支部マスタ 256 = 計 306）を **すべて緑のまま維持**。

### 5.2 新規追加テスト

#### 5.2.1 F7 編集

- name のみ編集 → マスタ更新
- yomi のみ編集 → マスタ更新
- name と yomi 両方編集 → マスタ更新
- name 空文字でエラー
- yomi 空文字 OK
- 編集後の同名重複検出（warn 通知）
- 編集後に過去参加者パネルが再描画される

#### 5.2.2 F7 削除（tombstone）

- 削除ボタン → `deleted=true`, `deleted_at` 設定
- 削除後はマスタタブから消える
- 削除後は過去参加者パネルからも消える
- LocalStorage に tombstone は残る（物理削除されていない）
- 削除された member の id を A-3 で復元しても tombstone のまま（A-3 では UI で復元不可）

#### 5.2.3 F8 エクスポート

- ダウンロードされる JSON が schema_version: 1
- tombstone 含む全 member が含まれる
- ファイル名が日付フォーマット
- 空マスタでもエクスポート可能（members: []）

#### 5.2.4 F8 インポート（上書き）

- 上書き選択 → 既存マスタが破棄される
- インポートされた member が反映される
- tombstone も復元される（A-3 では UI で復元不可だが、データとしては保持）

#### 5.2.5 F8 インポート（マージ）

- マージ選択 → 既存マスタに追加
- 同名重複は新規扱い（A-1 同じ）
- 既存と member_id が同じなら統合（last_attended は新しい方）
- 結果サマリ：読込 / 新規追加 / 既存統合 / スキップ

#### 5.2.6 F8 不正データ検証

- schema_version が違う → エラー
- members が配列でない → エラー
- 破損 JSON → エラー
- 個別 member のフィールド不足 → 補完

#### 5.2.7 クイックフィルタ

- 「最終参加日 直近」→ 全 member の最大 last_attended と一致するもの
- 「3ヶ月以内」→ today から 90 日以内
- 「常連 (5回以上)」→ attendance_count >= 5
- 排他選択動作（再タップで解除）
- 50音タブとの AND
- 検索との AND
- フィルタ + タブ + 検索の3条件 AND

### 5.3 性質テスト

- F7 編集で同じ id を維持（id 変更なし）
- F7 削除前後で `members.length` 変わらず（物理削除されない）
- F8 エクスポート → インポート（上書き） で完全に同じ状態に戻る
- F8 エクスポート → 削除 → インポート（マージ）で削除分が復活する

### 5.4 既存大会運営の非破壊（A-1/A-2 と同じ哲学）

- F7/F8/クイックフィルタを使わなければ A-2 と同じ動作
- 既存テスト 306 件すべて緑

## 6. 実装順（推奨フェーズ分け）

## 8. ChatGPT メタレビュー観点（依頼用）

このドキュメントを ChatGPT に渡す際、以下の観点でレビュー依頼します：

1. **データモデルの後方互換性**：A-1/A-2 マスタを壊さないか、A-3 で削除した member を A-2 が読んでも壊れないか
2. **F7 編集スコープの妥当性**：name + yomi のみ編集可能、参加履歴は読み取り専用の判断
3. **F7 削除の tombstone 方式**：物理削除しない判断、復元 UI なしで運用に支障ないか
4. **F8 戦略 Z（上書き + マージ）の妥当性**：UI が複雑になりすぎないか、ユーザーが誤って上書きしない設計
5. **クイックフィルタ3種の選定と排他選択**：複数選択（AND）の方が良いケースはないか
