## 1. 運用前提(本仕様の出発点)

本アプリはローカル DB を持たず localStorage のみ。本番直前は以下の運用を想定する:

1. 既存マスタをエクスポート(バックアップ)
2. データリセット(マスタ空)
3. JSON 取り込み(`20260412_participants.json`、22 名)
4. 過去参加者パネルで 22 名表示確認
5. 高橋一雄(本人) など必要対象のみ yomi / member / grade を F7 で補完
6. 5/10 当日: 過去参加者パネルから A 級・B 級振り分け

つまり Phase 2 は **「リセット」+「import」+「本番運用 runbook」** の 3 点セット。

### 1.1 Phase 2 着手条件 (整合性レビュー指摘反映)

以下すべてを満たした時点で Phase 2 着手可能とする:

- Phase 1 が main へマージ済み
- person schema に city フィールドが補完される(§2.1 正規化ルール込み)
- F7 編集モーダルで city の保存・再表示ができる
- backup / restore round-trip で city が保持される
- 既存 e2e 緑が維持されている

## 2. 取り込みデータ

- ファイル: `data/import/20260412_participants.json`(本仕様書と同時提供済)
- 22 名(A 級 18 + B 級 4)、city 付き、yomi / member / grade は空
- 共通: `last_played = "2026-04-12"`、`last_class = "A" or "B"`
- 特記: 大竹智也(A 級)は高校生 → grade 補完時「一般」、高橋一雄(B 級)は本人扱い

リセット後の取り込みのため、当初話に出した「name + last_played でマージ判定」は **不要**(リセットで既存全員が消えるため重複は発生しない)。

### 2.1 import 前検証条件 (ChatGPT Must Fix 4 反映)

import 実行前に以下をすべて検証し、1 つでも失敗したら全件ロールバックする:

- 配列長 = 22
- 全件 `last_played === "2026-04-12"`
- 全件 `last_class ∈ {"A", "B"}`
- A 級 18 名・B 級 4 名
- 全件 name 非空
- 全件 city が 20 文字以内
- yomi / member / grade は空文字許容
- unknown field は許容するが、保存時に既存 person schema へ正規化する(Phase 1 §2.1 と同一ルール)

## 3. Plan Mode で必ず確認すべき項目

着手時 Shift+Tab×2 で Plan Mode に入り、以下を読み取りで確認してから計画を出すこと:

- 既存に **リセット機能** があるか(マスタ全削除 UI、誤操作防止の有無)
- 既存に **JSON import 機能** があるか、ある場合のフォーマット仕様と挙動(マージ / 上書き / リセット後追加 のどれか)
- 既存に **export 機能**(バックアップ用) があるか、JSON 形式と localStorage キー名
- person オブジェクトのスキーマ(Phase 1 で追加した city 含めた完全形)
- **過去参加者パネル** の表示ソース(マスタ全件 / last_played 1 年以内 / その他絞り込み)

## 5. UI / 機能要件(ケース B〜C で実装する場合の最小仕様)

### 5.1 データリセット機能(無ければ追加) (ChatGPT Must Fix 1 反映)

リセット実行前のバックアップ取得を **機械的または runbook 上の必須条件** とする。最低でも下記 C 案、可能なら B 案を採用する:

- **A 案(理想)**: export 実行後のみリセットボタンを有効化する(セッション内 export 履歴管理)
- **B 案(推奨)**: リセット確認ダイアログ内に以下を表示する
  - 「バックアップ JSON をダウンロード済み」チェックボックス(必須)
  - 現在のマスタ件数表示
  - 最終 export 時刻表示(セッション内のみで可)
  - 二段階確認(チェックボックス + 「リセット」テキスト入力)
- **C 案(最低限)**: B 案実装が間に合わない場合、runbook で「バックアップファイル名・保存場所・件数を人間が確認してからリセットする」手順を必須化する

L-1〜L-4 安全境界準拠。誤操作防止の二段階確認は A〜C いずれの場合も必須。

**最終採用方針 (ChatGPT 再レビュー Should Fix 1 反映)**: 実装コストと本番日程を考慮し、**原則 B 案**、間に合わない場合のみ C 案へフォールバックする。A 案は本フェーズでは Out of Scope(セッション内 export 履歴管理は実装コスト過大)。

### 5.2 JSON 取り込み機能(無ければ追加) (ChatGPT Must Fix 2 反映)

**Phase 2 の import は「空マスタへの新規取り込み専用」とする**:

- マスタ管理画面に「JSON 読込」ボタン
- ファイル選択 → §2.1 検証 → プレビュー(取り込み件数・級別集計・**サンプル 3 件: A 級 1 件・B 級 1 件・city が長めのもの 1 件 を含むことを推奨**(ChatGPT 再レビュー Should Fix 2 反映))→ 実行
- **import 実行時に既存マスタが 1 件以上ある場合は、原則として実行を拒否し**「先にバックアップとリセットを実施してください」と表示する
- 検証失敗時: エラーメッセージで該当行を示し、**全件ロールバック**(部分適用しない)
- 成功時: localStorage 更新 + 過去参加者パネル即時反映

本フェーズではマージ import / 上書き import / 差分 import は **Out of Scope**。マージ判定不要を安全に成立させるため空マスタ専用に制限する。

## 6. 受け入れ条件 (Gate-level e2e)

1. リセット機能: §5.1 二段階確認 + バックアップ条件(A/B/C 案いずれか) OK でマスタが空になる
2. リセット機能: キャンセル / 確認失敗で何も起きない(誤操作防止)
3. import: 既存マスタが 1 件以上の状態で実行 → 拒否される(空マスタ専用制限)
4. import: 22 名すべて登録される(name / city / last_class / last_played 正確)
5. import: §2.1 検証通過時のみ適用、失敗時は全件ロールバック
6. import: city が Phase 1 §2.1 正規化ルール通りに保存される
7. 過去参加者パネルに 22 名表示される(A 級 18 / B 級 4)
8. F7 編集モーダルで **22 名全員について** yomi / member / grade を編集可能
9. **既存 e2e 緑維持**(Phase 1 マージ後の数値ベース)
10. **復元テスト (ChatGPT Should Fix 3 反映)**: バックアップ → リセット → 取り込み → 復元 の round-trip で、件数 + name + yomi + member + grade + city + last_class + last_played + その他既存フィールド すべてが **JSON stringify ベースの完全一致、または正規化後 deepEqual** で元通り

### 6.1 本番前の必須補完対象 (ChatGPT Should Fix 4 反映)

「全員編集可能」と「本番前に補完が必要な対象」を分ける:

- 全員編集可能: 受け入れ条件 8 のとおり、22 名分すべてで F7 補完できる
- **本番前必須補完**: 高橋一雄(本人、yomi / member / grade)、大竹智也(grade=「一般」)、その他事前判明している対象のみ
- 残りの参加者は本番中・本番後の F7 編集で随時補完

## 7. Plan B(本番リスクヘッジ): DevTools 直叩き運用 (ChatGPT Must Fix 3 反映)

ケース D で機能追加が間に合わない場合の保険。採用時は以下を **すべて必須条件** とする:

1. 実行前に localStorage の対象キー名を Claude Code が確認し、runbook に明記する
2. 実行前に `localStorage.getItem` の結果をファイル保存し、件数を確認する
3. setItem 用コードは Claude Code が生成し、人間が手修正しない
4. setItem 実行後、即座に画面リロードして 22 名表示を確認する
5. 復元用 setItem コード(バックアップ JSON からの再投入用)も同時に用意する
6. 当日朝ではなく、**可能なら前日(5/9)または事前リハーサルで一度実行する**

ガードなしの DevTools 直叩きは本番データ破壊リスクが高いため、上記 6 条件を欠かさないこと。Plan B 採用判断は Plan Mode 結果共有後に再相談。

## 8. リスク (P0)

- リセット誤操作で実データ消失 → §5.1 二段階確認 + バックアップ機械条件で対処
- import フォーマット不一致でデータ欠損 → §2.1 検証 + Plan Mode で既存仕様確認
- import 中の例外で部分適用 → 全件ロールバック必須
- 本番前日(5/9)実装 → リスク高、ケース D は §4 のとおり Plan B 優先
