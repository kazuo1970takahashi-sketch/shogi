# Phase 2 Mini 仕様書: 過去参加者 22 名取り込み + 本番前リセット運用

- 対象: shogi_v4.html(沼津支部運営ツール)
- 関連: Phase 1(city フィールド追加)マージ完了が前提、roadmap v21、Postmortem v2
- 目的: 5/10(日)月例将棋大会本番で過去参加者パネル(2026-04-12 参加 22 名)を A/B 振り分けに使える状態にする
- 区分: Mini Spec / Gate Review / **データ破壊リスク P0**

---

## 1. 運用前提(本仕様の出発点)

本アプリはローカル DB を持たず localStorage のみ。本番直前は以下の運用を想定する:

1. 既存マスタをエクスポート(バックアップ)
2. データリセット(マスタ空)
3. JSON 取り込み(`20260412_participants.json`、22 名)
4. 過去参加者パネルで 22 名表示確認
5. 高橋一雄(本人) の yomi / member / grade を F7 で補完
6. 5/10 当日: 過去参加者パネルから A 級・B 級振り分け

つまり Phase 2 は **「リセット」+「import」+「本番運用 runbook」** の 3 点セット。

## 2. 取り込みデータ

- ファイル: `data/import/20260412_participants.json`(本仕様書と同時提供済)
- 22 名(A 級 18 + B 級 4)、city 付き、yomi / member / grade は空
- 共通: `last_played = "2026-04-12"`、`last_class = "A" or "B"`
- 特記: 大竹智也(A 級)は高校生 → grade 補完時「一般」、高橋一雄(B 級)は本人扱い

リセット後の取り込みのため、当初話に出した「name + last_played でマージ判定」は **不要** になる(リセットで既存全員が消えるため重複は発生しない)。

## 3. Plan Mode で必ず確認すべき項目

着手時 Shift+Tab×2 で Plan Mode に入り、以下を読み取りで確認してから計画を出すこと:

- 既存に **リセット機能** があるか(マスタ全削除 UI、誤操作防止の有無)
- 既存に **JSON import 機能** があるか、ある場合のフォーマット仕様と挙動(マージ / 上書き / リセット後追加 のどれか)
- 既存に **export 機能**(バックアップ用) があるか、JSON 形式
- person オブジェクトのスキーマ(Phase 1 で追加した city 含めた完全形)
- **過去参加者パネル** の表示ソース(マスタ全件 / last_played 1 年以内 / その他絞り込み)

## 4. ケース分岐(Plan Mode 結果で確定)

| ケース | リセット | import | 実装スコープ |
|---|---|---|---|
| **A** | 既存 | 既存 | JSON フォーマットを既存に合わせる + runbook のみ |
| **B** | 既存 | なし | import 機能を最小実装(F7 上部に「JSON 読込」ボタン追加) |
| **C** | なし | 既存 | リセット機能を最小実装(誤操作防止ダイアログ必須) |
| **D** | 両方なし | - | 両方最小実装。本番リスク高、スコープ最小化判断必要 |

ケース D の場合は実装着手前に再相談。Plan B(下記§7)も検討する。

## 5. UI / 機能要件(ケース B〜D で実装する場合の最小仕様)

### 5.1 データリセット機能(無ければ追加)

- マスタ管理画面に「データリセット」ボタン
- 押下で確認ダイアログ:「マスタを全削除します。先にエクスポートでバックアップを取りましたか?」+ 二段階確認(チェックボックス + 「リセット」テキスト入力)
- L-1〜L-4 安全境界準拠

### 5.2 JSON 取り込み機能(無ければ追加)

- マスタ管理画面に「JSON 読込」ボタン
- ファイル選択 → 検証 → プレビュー(取り込み件数・サンプル 3 件)→ 実行
- 検証: 必須フィールド(name)+ 型チェック + 既存スキーマ準拠 + city フィールド長
- 失敗時: エラーメッセージで該当行を示す(ロールバックして部分適用しない)
- 成功時: localStorage 更新 + 過去参加者パネル即時反映

## 6. 受け入れ条件 (Gate-level e2e)

1. リセット機能: 二段階確認 OK でマスタが空になる
2. リセット機能: キャンセル / 確認失敗で何も起きない(誤操作防止)
3. import: 22 名すべて登録される(name / city / last_class / last_played 正確)
4. import: city が Phase 1 仕様通り保持される
5. import: 不正 JSON で部分適用されない(全件ロールバック)
6. 過去参加者パネルに 22 名表示される(A 級 18 / B 級 4 でフィルタ可能なら確認)
7. F7 編集モーダルで 22 名分の yomi / member / grade 補完ができる
8. **既存 e2e 緑維持**(Phase 1 マージ後の数値ベース)
9. **バックアップ → リセット → 取り込み → 復元** の round-trip で全件元通り

## 7. Plan B(本番リスクヘッジ): DevTools 直叩き運用

ケース D で機能追加が間に合わない場合の保険:

- DevTools コンソールで `localStorage.setItem` を直接実行
- 事前に既存マスタを `localStorage.getItem` で控え(バックアップ)
- runbook に手順を残し、当日 5/10 朝に髙橋さん自身が実行
- ただし運用としては脆く本番後速やかに正式機能化が望ましい

Plan B 採用判断は Plan Mode 結果共有後に再相談。

## 8. リスク (P0)

- リセット誤操作で実データ消失 → 二段階確認 + 直前バックアップ運用で対処
- import フォーマット不一致でデータ欠損 → Plan Mode で既存仕様に厳密に合わせる
- import 中の例外で部分適用 → 全件ロールバック必須
- 本番前日(5/9)実装 → リスク高、Plan Mode 後にスコープ縮小判断
