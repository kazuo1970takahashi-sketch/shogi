## 2. ケース判定(仕様書 §4)

| ケース | リセット | import | 判定 |
|---|---|---|---|
| A | 既存 | 既存 | × |
| **B** | 既存 | **なし** | × |
| **C** | **なし** | 既存 | △ 既存 import は形式不一致のため流用困難 |
| D | 両方なし | - | × |

**結論**: 厳密には **ケース C** だが、既存 import の形式制約(id 必須、members オブジェクト形式、空マスタ専用制限なし)により**既存 import を流用するとデータ破壊リスクが高い**。

→ **採用方針: ケース C 改 = リセット機能新規実装 + Phase 2 専用 import 機能新規実装**(既存 import モーダルは不変)

## 5. 承認ポイント(髙橋さんへ)

以下 3 点について承認/修正指示をお願いします:

### 5.1 ケース判定: **C 改(リセット新規 + Phase 2 専用 import 新規)**

仕様書 §4 ケース判定:
- 既存リセット機能なし(L361 「resetAll() ではマスタを消さない」明記)
- 既存 import 機能あり(`applyOverwriteImport` / `applyMergeImport`)だが、形式が `{members:[]}` 必須 + `id` 必須 + 空マスタ専用制限なし → **取り込みデータ形式と不一致**

→ 既存 import を流用せず、Phase 2 専用 import 機能を新規実装する判断が筋。
→ **採用方針: ケース C 改**(リセット新規 + Phase 2 専用 import 新規、既存 import モーダルは不変)

### 5.2 リセット §5.1 採用案: **B 案**

- A 案(セッション内 export 履歴管理)は実装コスト過大、本フェーズでは Out of Scope
- **B 案採用**: バックアップ済チェックボックス(必須)+ 件数表示 + 「リセット」テキスト入力 + 二段階確認
- C 案(runbook のみ)は B 案実装が間に合った場合は不採用

### 5.3 既存「マスタをインポート」との整合方針

**既存 import モーダル(overwrite/merge)は不変、Phase 2 専用 import を別ボタンで追加**:

- 理由 1: 取り込み形式が配列形式(現状の JSON ファイル)で、既存 import の `{members:[]}` 形式と非互換
- 理由 2: 空マスタ専用制限を既存実装に載せると merge / overwrite の挙動も縛られる(Out of Scope への波及)
- 理由 3: 「📥 22 名取込(5/10 大会用)」と専用名称にすることで操作の意図が明確、本番後に削除/隠す方針も取りやすい(将来的に Out of Scope 扱い)

承認後、Step 1〜3 を連続実行します。
