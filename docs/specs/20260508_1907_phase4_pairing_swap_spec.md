## 1. 背景

5/8 実機運用検証で発覚: ペアリング後の「対戦相手の変更」モーダル(`<h3>対戦相手の変更</h3>` / 保存ボタン `chg-save`)が **常に重複エラーで弾かれ、実質機能していない** 状態。

調査結果(2026-05-08 grep + git log):
- swap / 入れ替え / alternative 系ロジックは shogi_v4.html 内ゼロヒット
- 初回大規模 commit `5dd90ad`(2026-04-21)から **1 名置換 replace UI** として実装、swap 機能は元から未実装
- 過去 6 ヶ月で swap 系 commit なし
- 既存 helper: `pairHasRematch`(再戦検知)/ `getDuplicatePlayersInPairings`(重複検知)→ swap 実装で再利用可

L3199 の削除阻害メッセージ「先に『対戦相手変更』で別の選手に差し替えてから削除してください」から、現状 UI は **削除前の代替選手立て(欠員補充)用途** で設計されたと推定。任意の組み合わせ変更には設計されていない。

## 2. 目的

「対戦相手の変更」UI の動作仕様を **replace のみ → replace + swap の自動判別** に変更し、運営者の期待する任意の組み合わせ変更を可能にする。

## 3. ユーザーストーリー

**主**: 大会運営中、ペアリング画面で「○○ の対戦相手を △△ に変えたい」と思った時、△△ が既に別ペアに居ても 2 ペア同時 swap で対応できる。再戦が発生する場合はエラーで防がれる。

**従**: 削除予定選手の代替として欠員を補充する従来用途も維持される(replace 動作)。

## 4. 機能仕様

### 4.1 入口

- 対局管理タブで対象ペアの「変更」ボタン押下
- **prerequisite チェック(モーダル開く前)**: 対象ペアが winner 入力済みの場合、モーダル開かずエラー表示「結果入力済みのため変更できません」(下記 §4.5)
- モーダル `<h3>対戦相手の変更</h3>` 表示
- 対象選手 = ボタンを押したペアの片方
- 変更先選手選択 UI = 同クラス全選手から対象選手と既ペア相手を除外(従来仕様維持)

### 4.2 確定動作(新規ロジック)

「変更を保存」(`chg-save`)押下時、変更先選手 X の状態で 2 系統に分岐。

**X の所属判定範囲**: **現在編集中の同クラス・同ラウンド内の pairings に限定**(過去ラウンドの結果 / 別クラスは判定対象外)。

#### 系統 A: replace(従来動作)
- X が **現在ラウンドのどのペアにも所属していない**(過去ラウンドのみ存在 / 削除済み / BYE 等)場合
- 対象ペア(○○-▲▲)の ▲▲ を X に置換 → (○○-X)
- 再戦チェック: `pairHasRematch(○○, X)`(**確定済みの過去ラウンドのみが探索対象**、現在編集中ラウンドの旧ペア状態は除外、§10 で実装前確認)→ true なら エラー表示
- OK なら 1 ペアのみ更新

#### 系統 B: swap(新規動作)
- X が **現在ラウンド内の別ペア(X-Y)に所属している**場合
- **prerequisite**: X のペア(X-Y)が winner 未入力であること(winner 入力済みなら swap 拒否、§4.5)
- 自動 swap 提示: 元のペア(○○-▲▲)と X のペア(X-Y) → swap 後(○○-X)と(▲▲-Y)
- 再戦チェック: `pairHasRematch(○○, X)` AND `pairHasRematch(▲▲, Y)` の両方 false が必要(過去ラウンドのみ対象、§10)
- どちらか true なら エラー表示「swap で再戦が発生します。別の選手を選んでください」(キャンセル)
- OK なら 2 ペア同時更新

### 4.3 確認ダイアログ

swap 動作時のみ、確定前に確認ダイアログ表示:
```
swap を実行します。
  (○○-▲▲) → (○○-X)
  (X-Y)   → (▲▲-Y)
よろしいですか?  [OK / キャンセル]
```

replace 動作時は従来通り即時実行(確認なし)。

### 4.4 状態更新

- swap / replace 確定後、`state.results[cls][round][i]` のペア情報を更新
- 再戦履歴(`history` / pairings 配列)は更新不要(まだ対局していないラウンド)
- localStorage 自動保存(従来通り)
- `renderTournament` 再描画

### 4.5 prerequisite チェック(全体)

モーダル開く前 / 保存時の事前チェック:

1. **対象ペアの winner 入力チェック**(モーダル開く前): 対象ペアが winner 入力済み → モーダル開かない、エラー表示「結果入力済みのため変更できません」
2. **swap 相手ペアの winner 入力チェック**(系統 B、保存時): X のペア(X-Y)が winner 入力済み → swap 拒否、エラー表示「相手ペアが結果入力済みのため swap できません」
3. **同ペア内 swap 拒否**(保存時): X が対象ペア(○○-▲▲)の ▲▲ 自身 → 保存時拒否「同じペア内では変更できません」(UI 上選べないが念のため防御)
4. **swap 後の重複チェック**(保存時): swap 後の同一ラウンド内に重複選手が発生していないことを `getDuplicatePlayersInPairings` で検証(防御層、通常は発生しない)
5. **再戦チェック対象**: `pairHasRematch` の探索対象は **「確定済みの過去ラウンドのみ」**、現在編集中ラウンドの旧ペア状態は除外(§10 で実装前確認)

## 5. 非機能仕様

- 既存 e2e 702 件すべて緑維持
- 新規 e2e 5〜10 件追加(下記 §7)
- 既存 helper(`pairHasRematch` / `getDuplicatePlayersInPairings`)再利用、関数追加最小化
- 関数追加: 1 つ(`findPairContainingPlayer(cls, round, playerId)` のような選手所属ペア検索)が想定範囲

## 6. 不変条件(維持)

- **INV-PA-1**: 同じ相手と複数回対戦しない(swap 後ペアでも再戦チェック)
- **INV-PA-2**: 各ペアは 2 名で構成、欠員/重複不可
- **INV-PL-1**: 同一名重複不可
- swap 確定後の状態は、自動ペアリング直後と同じ整合性を持つ(検証可能)

## 7. テスト戦略(e2e)

新規追加 e2e ケース:

1. **swap 成功**: A-B / C-D の B を C に変更 → A-C / B-D に更新確認
2. **swap で再戦衝突**: 過去ラウンドで A-C 対戦済 → swap 試行でエラー表示
3. **replace 成功**(従来動作): 削除済選手の代替で空席を埋める → 1 ペアのみ更新
4. **swap 確認ダイアログキャンセル**: ダイアログで「キャンセル」→ 状態変化なし
5. **swap で過去結果に影響なし**: 過去ラウンド(対局済み)のペアは変更されない
6. **winner 入力済みペアで変更ボタン**: モーダル開かず、エラー表示
7. **swap 相手ペアが winner 入力済み**: swap 拒否、エラー表示
8. **swap 後の同ラウンド重複なし**: `getDuplicatePlayersInPairings` で 0 件確認

既存 e2e 702 件は影響なし(replace 動作は従来通り維持)。

## 8. 範囲外(永続見送り or 後 Phase)

### 機能スコープ
- **クラス間 swap**: A 級 ⇄ B 級の swap は対象外(クラス変更は別 UI)
- **過去ラウンド swap**: 対局済みラウンドのペア変更は対象外(現在進行中ラウンドのみ)
- **3 ペア以上の循環 swap**: 2 ペア固定、3 ペア循環は対象外(複雑度大幅増、運用頻度低い想定)
- **swap 候補の自動探索**: ユーザーが明示的に変更先を指定する前提、システムが代替候補を提示する機能は対象外
- **winner 入力済みペアの巻き込み swap**: 明示的に禁止(prerequisite で拒否、§4.5)
- **BYE / 運営者 / 空席を含む特殊 swap の一般化**: 通常選手間の swap のみ対応、BYE/空席は従来 replace 動作に委ねる

### 実装スコープ
- **自動ペアリングロジック本体の変更**: 自動生成アルゴリズム(バックトラッキング / repair)には触れない
- **UI 全面刷新**: 既存モーダル `<h3>対戦相手の変更</h3>` を流用、UI/UX 改善は別 Phase
- **対局済み結果 winner の修正**: 別 UI(editPastResult)で対応済み、本 Phase の範囲外

## 10. 実装手順案(Plan Mode 用ヒント)

1. 現在の `chg-save` クリックハンドラの所在確認(grep `chg-save` で関数特定)
2. **既存 `pairHasRematch` の探索対象を確認**: 過去ラウンドのみか、現在ラウンド暫定 pairings/results も含むか。後者なら `currentRound` を除外できる引数追加 or 内部分岐を検討(**実装前必須確認事項**)
3. 変更先選手 X の所属ペア検索ヘルパ実装(`findPairContainingPlayer(cls, round, playerId)` のような名称)

5. クリックハンドラを 2 分岐(replace / swap)に拡張
6. swap 分岐で再戦チェック(両ペア)+ 確認ダイアログ + 2 ペア更新
7. swap 後の `getDuplicatePlayersInPairings` 防御層チェック追加
8. e2e ケース追加 + 全件 green 確認
