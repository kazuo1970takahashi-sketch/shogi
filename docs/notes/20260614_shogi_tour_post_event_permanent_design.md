## 1. 背景

本日の大会運営後、リアルタイム表示は好評だった。
一方で、以下の追加要望が出ている。

- スマホでリアルタイム閲覧したい
- 実際の対戦表に近い ○× の星取表で見たい
- 氏名にふりがなを付けたい
- ふりがなは Word / Excel のルビのように、氏名の真上に小さく表示したい
- 対局管理画面の名前ボックスに氏名が収まらない
- PDF 出力ファイル名を大会名ベースにしたい
- PDF 内タイトルに対象月・開催日・大会名を出したい
- 参加者マスタをスプレッドシートのように一覧編集したい
- 過去の月例会結果を蓄積し、過去対戦成績や年間順位に発展させたい
- スマホ閲覧用 URL を、参加者が手入力しやすい短い URL にしたい

このうち、**即日対応（LIVE-MOBILE-001）で着手するもの**と、**本メモで恒久対応として整理するもの**を切り分ける。

### 1.1 即日対応側（本メモの対象外。LIVE-MOBILE-001 で実装中）

- スマホでのリアルタイム閲覧（暫定運用）
- ○× 星取表の表示
- 対局管理画面の名前ボックスに氏名が収まる表示改善
- PDF 出力ファイル名を大会名ベースにする
- PDF 内タイトルに対象月・開催日・大会名を出す

### 1.2 本メモの対象（後続恒久対応 4 テーマ）

この設計メモでは、即日実装対象ではない以下 4 テーマを恒久対応として整理する。

1. **ふりがな ruby 表示と参加者マスタ ID**
2. **参加者マスタのスプレッドシート形式一覧編集**
3. **月例会結果アーカイブ／大会履歴 DB 化**
4. **スマホ閲覧用 URL の恒久対応**

> 補足: 即日対応の「ふりがな表示」「星取表のスマホ閲覧」は暫定運用としても成立する一方で、ふりがなを**正式なデータ項目として保持**し、**同姓同名に耐え**、**過去大会の履歴を崩さない**ためには、参加者マスタ ID とスナップショットを前提とした恒久設計が必要になる。本メモはその土台を提供する。

## 2. 基本方針

後続 4 テーマ全体に共通して、以下を恒久設計の前提とする。

- **暫定対応ではなく恒久対応として設計する。** 当日その場しのぎではなく、年間運営・複数年運用に耐える構造を最初に定義する。
- **氏名だけで参加者を紐づけない。** 表示名は変わり得るため、紐づけは安定した ID で行う。
- **同姓同名に耐える。** 表示が同一でも別人として扱えるよう、`participantMasterId` で識別する。
- **マスタ変更後も過去大会結果が勝手に変わらない。** 過去大会の表示は当時の値を再現する。
- **大会時点の氏名・ふりがな等はスナップショットとして保持する。** 集計用 ID と表示用スナップショットを分離する。
- **既存保存データとの互換性を壊さない。** 既存の大会保存 state / 候補マスタの形式は変更しない。新項目は省略可能（optional）として追加する設計とし、無くても読み込める。
- **既存の順位計算・勝敗入力・クラス別運営ロジックを不用意に変更しない。** 新機能は既存ロジックの外側に重ねる（read 層・表示層・別ストア）。
- **画面ごとにバラバラ実装せず、共通 helper / 共通データ構造を使う。** ふりがな表示・スナップショット生成・エスケープ等は共通化する。
- **スマホ閲覧は read-only を原則とする。** 閲覧用 UI からは編集・保存・勝敗入力を出さない。

> データモデルの現状（前提）: 候補マスタ（localStorage `shogi_branch_master`, `members[]`）と当日大会 state（`shogi_v4`, `players/pairings/results/started`）は別キーで分離されている。本設計はこの分離を保ったまま、**マスタ側に正式項目を足し**、**大会 state 側に当時スナップショットを足す**方向で拡張する（いずれも後続 PR・本メモでは実装しない）。

## 3. ふりがな ruby 表示の恒久対応

### 3.1 データ方針

参加者マスタにふりがなを正式項目として保持する。

**参加者マスタ（master）想定項目:**

| 項目 | 役割 | 備考 |
| --- | --- | --- |
| `participantMasterId` | 一意な内部 ID | ユーザー編集不可。紐づけ・集計の主キー。 |
| `name` | 氏名（表示名） | 変更され得る。 |
| `kana`（別名 `furigana`） | ふりがな | ruby の `rt` に使う。未入力可。 |
| `branchName` | 支部名 | |
| `memberType` | 支部員／一般 | 既存の会員区分と整合させる。 |
| `feeType` | 会費区分 | |
| `active` | 有効／無効 | 削除ではなく無効化に使う（4.5 参照）。 |
| `note` | 備考 | |
| `createdAt` | 作成日時 | |
| `updatedAt` | 更新日時 | 一覧編集での更新検知に使う。 |

> 命名は既存コードの語彙に合わせて後続 PR 設計時に最終決定する。`kana` / `furigana` はどちらか一方に統一する（本メモ内では `kana` を代表名として扱う）。

大会参加者には、**マスタ ID と大会時点スナップショット**を保持する。

**大会参加者（tournament player）想定項目:**

| 項目 | 役割 | 備考 |
| --- | --- | --- |
| `tournamentPlayerId` | 大会内の一意 ID | |
| `tournamentId` | 所属大会 | |
| `participantMasterId` | マスタへの参照 | 集計・同一人物判定に使う。 |
| `nameSnapshot` | 大会時点の氏名 | 表示用。当時の値を再現。 |
| `kanaSnapshot` | 大会時点のふりがな | 表示用。 |
| `branchSnapshot` | 大会時点の支部 | 表示用。 |
| `className` | クラス | 既存のクラス別運営に整合。 |
| `entryNo` | 参加番号 | |

**この分離を採る理由:**

- 同姓同名対策（表示が同じでも `participantMasterId` で別人と識別できる）。
- マスタ修正後も過去大会結果を維持するため（過去大会は `*Snapshot` を表示する）。
- 後からふりがなを整備しても、過去大会の表示を勝手に変えないため。
- 年間順位や過去対戦成績に発展させるため（`participantMasterId` で横断集計できる）。

### 3.2 ruby 表示方針

ふりがなは全機能で **ruby 表示に統一**する。

要件:

- Word / Excel のふりがなルビに近い見た目にする。
- ふりがなは**氏名の真上**に小さく表示する。
- 氏名の**下に別行表示する形式ではない**。
- HTML では `<ruby>氏名<rt>ふりがな</rt></ruby>` を使う。
- `rt` の `font-size` は氏名の **50〜60% 程度**を目安にする。
- `ruby-position: over` を指定する。
- スマホ星取表でもルビが大きすぎて**行高を圧迫しない**ようにする（星取表側では `rt` をさらに小さめに、または行高を固定して overflow を抑える）。
- PDF 出力時も可能な範囲で同じルビ表示を維持する（印刷 CSS でも `ruby` が潰れないようにする。崩れる環境向けの代替表示は 8 章リスク参照）。

**設計イメージ（実装ではない / `shogi_v4.html` には未追加）:**

```html
<!-- 生成されるべき DOM 構造の例 -->
<ruby class="player-name">架空 太郎<rt>かくう たろう</rt></ruby>
```

```css
/* スタイルの方向性の例。星取表では .compact 側でさらに抑制 */
.player-name { ruby-position: over; }
.player-name rt { font-size: 0.55em; line-height: 1; }
.scoreboard .player-name rt { font-size: 0.5em; }
```

### 3.3 共通 helper 方針

画面ごとに個別で ruby HTML を組み立てない。**共通 helper** を用意する。

想定 helper（シグネチャは設計イメージ）:

- `renderPlayerNameWithRuby(name, kana)` … エスケープ済みの ruby HTML 文字列を返す。
- `createPlayerNameRubyElement(name, kana)` … DOM API で `ruby` / `rt` 要素を生成して返す（textContent 経由で自動エスケープ）。
- `escapeHtml(value)` … HTML エスケープ共通関数。

**セキュリティ方針:**

- `name` / `kana` は必ずエスケープする。
- `innerHTML` に未エスケープ文字列を直接流さない。
- 可能であれば DOM API（`document.createElement` + `textContent`）で `ruby` / `rt` 要素を生成する（XSS 経路を構造的に断つ）。
- PDF 出力時も同じエスケープ方針を使う（出力経路ごとに別実装にしない）。
- `kana` が未入力（空 / null）の場合は `rt` を生成せず、氏名のみ表示する（空 `rt` を出さない）。

### 3.4 対象画面

ruby 表示は以下すべてで共通 helper を経由する。

- 参加者一覧
- 対局管理
- 順位表
- スマホ閲覧用星取表
- PDF 出力
- 報告書
- 将来の大会履歴表示

### 3.5 移行方針

- 既存参加者データに `kana` が無い場合は**氏名のみ表示**する。
- ふりがな未入力でも画面を崩さない（`rt` を出さない・行高を保つ）。
- 既存データの読込互換を維持する（`kana` / `*Snapshot` が無い既存 state もそのまま読める）。
- `kana` 未登録者を後から一覧編集（4 章）で補完できるようにする。
- 移行は「既存 → 新項目を optional 追加」の一方向のみ。既存項目の意味は変えない。

## 4. 参加者マスタのスプレッドシート形式一覧編集

### 背景

大会当日は、ふりがなが未入力または不完全なまま参加者マスタに登録されるケースが多い。
後日まとめて整備できる一覧編集 UI が必要。

### 4.1 UI 方針

- 参加者マスタを**表形式で一覧表示**する。
- スプレッドシートのように**一覧上で直接編集**できる。
- 1 人ずつ詳細画面を開かなくても修正できる。
- ふりがな未入力者を見つけやすくする。
- 編集前後の**差分**が分かるようにする。
- 保存前に確認できるようにする。
- ID は編集不可または内部管理とする。
- 誤操作で `participantMasterId` を壊せないようにする。

### 4.2 表示・編集候補項目

| 列 | 編集可否 | 備考 |
| --- | --- | --- |
| `participantMasterId` | 不可（表示のみ／内部管理） | 壊せないようにする。 |
| 氏名 | 可 | |
| ふりがな | 可 | 未入力フィルタ対象。 |
| 支部 | 可 | |
| 支部員／一般 | 可 | |
| 会費区分 | 可 | |
| 備考 | 可 | |
| 有効／無効 | 可 | 既定は有効。無効化で「削除」を代替。 |
| `updatedAt` | 不可（自動更新） | 保存時に更新。 |

### 4.3 便利機能候補

- ふりがな未入力フィルタ
- 支部別フィルタ
- 有効／無効フィルタ
- 氏名検索
- 重複候補検出
- 同姓同名候補検出（氏名一致だが `participantMasterId` が別のものを警告）
- 変更済み行のハイライト
- 一括保存
- 取消
- CSV import / export

### 4.4 保存方針

- マスタ編集は**参加者マスタのみ**を更新する。
- 過去大会の `nameSnapshot` / `kanaSnapshot` は変更しない。
- マスタ更新によって**過去大会結果の表示が勝手に変わらない**ようにする。
- 将来的に「過去大会スナップショットを明示的に補正する」機能を作る場合は、**別権限・別操作**にする（通常のマスタ編集と混在させない）。

### 4.5 安全性

- ID はユーザー編集不可。
- **削除ではなく `active=false` の無効化**を基本にする（過去大会からの参照を切らない）。
- 保存前に変更差分を表示する。
- ふりがな・氏名・支部名などは表示時にエスケープする（3.3 の `escapeHtml` を共通利用）。
- CSV import 時も危険文字や列不足に対応する（数式インジェクション対策として先頭 `= + - @` を無害化、列欠落時はデフォルト補完または行スキップ＋警告）。

## 5. 月例会結果アーカイブ／大会履歴 DB 化

### 背景

「過去の対戦成績」とは、**今大会内の履歴ではなく、過去の月例会結果を蓄積していく**意味である。
将来的に以下へ発展させる。

- 参加者別の過去成績
- 直接対戦成績
- 年間順位
- 年間ポイント
- 月例会別順位推移
- クラス別成績
- 表彰・昇級降級判断の補助

### 5.1 基本データ構造

| `name` | 大会名 |

| `targetYear` | 対象年 |

| `heldDate` | 開催日 |
| `type` | 種別（月例会／特別大会 等） |
| `createdAt` | |

**tournamentPlayers（大会参加者・結果）:**

| 項目 | 備考 |

| `participantMasterId` | 同一人物の通算集計キー |
| `nameSnapshot` | 表示用・当時の氏名 |

> `*PlayerId` は `tournamentPlayers.id`（大会内 ID）を指し、そこから `participantMasterId` に辿れる。これにより「表示は当時の名前」「集計は人物単位」を両立する。

### 5.2 スナップショット方針

- 大会終了時点の結果を保存する。
- 参加者名・ふりがな・支部・クラス・番号は**大会時点スナップショット**として保存する。
- マスタ変更後も過去大会結果が勝手に変わらないようにする。
- `participantMasterId` により、同一人物の通算成績を集計できるようにする。

### 5.3 年間順位・年間ポイント方針

- 年度単位で対象大会を集計する。
- 月例会ごとの順位・勝数・点数を集約する。
- 欠席・途中参加・クラス変更を扱える余地を残す（参加した大会のみ集計、未参加は欠損として扱う）。
- 同点時の順位決定ルールは別途定義する（既存の大会内 tie-break とは別レイヤの年間 tie-break を後続で定義）。
- 年間表彰や昇級降級判断への発展を想定する。

### 5.4 直接対戦成績方針

- `games` に `participantMasterId` へ辿れる `playerId` を保存する。
- 過去大会を横断して A vs B の対戦履歴を検索できるようにする。
- 同姓同名でも `participantMasterId` により識別する。
- `nameSnapshot` は表示用、`participantMasterId` は集計用と分ける。

### 5.5 保存・出力方針

- 既存の大会保存データを壊さない（既存 `shogi_v4` state はそのまま。アーカイブは別ストア／別ファイルとして派生させる）。
- まずは**アーカイブ用エクスポート／インポート**から開始してもよい（DB 化の前段として JSON 書き出し・取り込み）。
- 将来的に DB 化・クラウド保存・短い閲覧 URL（6 章）と接続できるようにする。
- 個人情報や実データの取扱いに注意する。
- repo に実名データや実大会データを入れない（テスト・サンプルは完全架空のみ）。

## 6. スマホ閲覧用 URL の恒久対応

### 背景

スマホ閲覧機能を本格運用するには、参加者がブラウザに直接入力しやすい**短い URL** が必要。
`file://` や IP アドレス形式では現場案内しづらい。
QR コード案内にも向かない。

### 6.1 恒久方針

- 独自ドメイン、または分かりやすいサブドメインを用意する。
- スマホ閲覧専用 URL を短くする。
- 運営用画面と閲覧用画面を**URL 上でも分離**する。
- QR コード案内にも使いやすい URL にする。
- 閲覧専用 URL では編集・保存・勝敗入力を出さない。
- **read-only を原則**にする。

### 6.2 URL 例（いずれも例示。ドメイン取得・公開は本メモの対象外）

- `https://shogi-tour.app/live`
- `https://shogi-tour.app/numazu/live`
- `https://numazu-shogi.jp/live`
- `https://live.numazu-shogi.jp`
- `https://shogi.example.com/live`

### 6.3 将来拡張

- 支部別 URL
- 大会別 URL
- 月例会別 URL
- 期間限定公開
- パスコード付き閲覧
- 会場内限定公開
- QR コード発行
- 運営者用 URL と閲覧者用 URL の分離
- 閲覧専用 API / データ同期方式

### 6.4 注意点

- 今日のローカル HTML 運用では恒久 URL 化は行わない。
- `file://` 形式は他のスマホから直接閲覧できない。
- IP アドレス形式は一時確認には使えるが、恒久運用には向かない。
- 公開時は個人情報、閲覧範囲、公開期間、パスコード要否を検討する。

> 現状の GitHub Pages 公開元は `production` ブランチ（`index.html` / `shogi_v4.html` の最小公開）。恒久 URL 化を検討する場合も、公開反映・ドメイン・ホスティング設定はユーザー明示承認のもとで別途行う（本メモでは設定変更を行わない）。

## 7. 実装順序案

後続対応の推奨実装順序。各段は独立 PR を想定し、前段の共通データ構造／helper を後段が利用する。

1. **参加者マスタ ID / `kana` 項目 / ruby helper 設計**（データ土台 + 共通 helper）
2. **スプレッドシート形式マスタ編集 UI**（`kana` 等を後日整備できる入口）

## 8. Risk / 注意点

- **氏名だけで紐づけると同姓同名で破綻する** → `participantMasterId` を主キーに、表示名はスナップショット。
- **マスタ変更で過去大会表示が変わると履歴として信頼できない** → 過去大会は常に `*Snapshot` を表示。マスタ補正は別操作。
- **ふりがな未入力でも画面が崩れないようにする** → `kana` 空時は `rt` を出さず氏名のみ。
- **ruby 表示はスマホ星取表で行高を圧迫しやすい** → 星取表は `rt` をさらに小さく／行高固定で overflow 抑制。
- **PDF 出力時に ruby が崩れる可能性がある** → 印刷 CSS で検証。崩れる環境向けに「氏名＋括弧書きふりがな」等の代替表現をフォールバックとして用意する余地を残す。
- **一覧編集 UI は誤編集・一括保存ミスに注意** → 保存前差分表示・取消・ID 編集不可・無効化主体。
- **履歴 DB 化は個人情報・実データ取扱いに注意** → repo には架空データのみ。実データは端末ローカル／別ストア。
- **短い URL 公開時は閲覧権限・公開期間・パスコード要否を検討する** → read-only 限定、運営 URL と分離。
- **repo に実データを入れない** → テスト・サンプル・fixture・コメントすべて完全架空（`架空 …` / `Dummy …` / `example.invalid` / `synthetic`）。
