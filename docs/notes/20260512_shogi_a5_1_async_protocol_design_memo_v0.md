## 1. 目的

SHOGI-TOUR の A-5.1 として、大会運営中における「どこまで不整合を許し、どこで整合性を取るか」を体系化する。

具体的には以下の3つを両立する非同期運営プロトコルを設計する。

- 大会進行を止めない柔らかさ
- データを壊さない厳格さ
- 現場運営文化と整合する自然さ

これは A-5.0 で確立した思想「人間系 eventual consistency」を、実装入口に接続できる境界定義レベルまで降ろす作業である。

## 3. 非同期運営プロトコルの定義

### 3.1 SHOGI-TOUR における定義

非同期運営プロトコルとは、大会運営において、複数の役員と複数のデータ操作を「常時完全同期」ではなく「フェーズ別の同期点」で整合させる運営モデルである。

各操作は自分のタイミングで進められ、最終的に同期点で整合する。

### 3.2 なぜ完全同期ではないか

完全同期（常時整合保証）は以下を要求する。

- すべての操作者が同一画面を見ている
- すべての入力が即時反映される
- 不整合状態を一切許さない

これは沼津支部の現実と矛盾する。

- 役員4名（幹事2 + 操作1 + 補佐1）が異なる作業を並行
- 紙とアプリの二系統で運用
- 対局中・受付中・集計中など複数フェーズが同時進行
- 「終わった人から」「気づいた人が」進める文化

完全同期を強制すると運営が止まる。

### 3.3 同期点設計の必要性

ただし、何もかも非同期にすると以下が起きる。

- 「壊れた状態」と「途中状態」の区別が消える
- データ破壊が事後発見になる
- 復旧不能な事故が起きうる

そのため明示的な同期点を設け、その瞬間に整合性を確認する。
同期点間は不整合許容、同期点で hard check という二段構えにする。

### 3.4 冒頭原則

A-5.1 設計の全章を貫く太い原則：

**published 後は rollback ではなく correction**

現場に出た情報（対局カード配布後・次ラウンドカード配布後・勝敗発表後など）は、戻すのではなく訂正として扱う。
全体 rollback を許すと現場が混乱する。
個別訂正を優先する。

## 4. rollback 段階モデル

A-5.0 で導入した5段階の状態機械を、A-5.1 で詳細化する。

### 4.1 draft

まだ現場に出ていない状態。

| 項目 | 内容 |
|---|---|
| 例 | 受付入力中、参加者情報入力中、対局組み合わせ作成前、カード生成前 |
| rollback | 自由 |
| 編集 | 自由 |
| 保存失敗時 | 警告のみ、進行は止める |
| UI示唆 | 編集可能を明示（ペンアイコン等） |

### 4.2 validated

システム上の整合確認済み、まだ現場配布なし。

| 項目 | 内容 |
|---|---|
| 例 | 受付完了、クラス確定、クラス別採番済み、組み合わせ生成済みカード未配布 |
| rollback | 可能だが影響範囲確認が必要 |
| 編集 | 修正可能（要確認モーダル） |
| 保存失敗時 | rollback推奨 |
| UI示唆 | 「確定前」表示、変更時は影響範囲を表示 |

### 4.3 published

現場に出た状態。

| 項目 | 内容 |
|---|---|
| 例 | 対局カード配布済み、参加者着席、1局目開始、次ラウンドカード配布済み |
| rollback | 原則不可 |
| 編集 | 訂正・差し替え・個別修正のみ |
| 保存失敗時 | 現場告知して訂正 |
| UI示唆 | 「現場展開済み」表示、編集は correction フローへ |

### 4.4 corrected

published 後に訂正された状態。

| 項目 | 内容 |
|---|---|
| 例 | 氏名訂正、会費訂正、勝敗訂正、対局カード差し替え |
| rollback | 不可（訂正履歴として残す） |
| 編集 | 再訂正のみ可 |
| 保存失敗時 | 訂正失敗ログ、再試行 |
| UI示唆 | 「訂正済み」表示 |

#### 訂正ログの段階設計

完全な append-only 監査ログは実装重量が大きいため、段階的に整備する。

| レベル | 内容 | A-5.1 対象 |
|---|---|---|
| Minimum | 訂正後の現在値 + 画面上の警告/メモ | A-5.1 で実装 |
| Standard | 訂正時刻・対象・変更前後を correction log として保存 | A-5.1 後半 or A-5.2 |
| Future | append-only の正式監査ログ | A-5.2以降 |

A-5.1 の初版実装は Minimum レベルで十分。
Standard / Future へは段階的に拡張する。

### 4.5 archived

大会終了後の保存・履歴化状態。

| 項目 | 内容 |
|---|---|
| 例 | 成績確定、参加履歴確定、会員マスタ反映済み、大会データ保存済み |
| rollback | 不可 |
| 編集 | 履歴付き修正のみ |
| 保存失敗時 | 大会終了処理を完了させない |
| UI示唆 | 「アーカイブ」表示、編集は管理者操作 |

## 5. commit 点一覧

### 5.1 commit 点定義

各 commit 点は以下の5要素 + 重さで定義する。

| 要素 | 内容 |
|---|---|
| イベント | 何が起きた時か |
| commit条件 | 何が満たされたら成立か |
| rollback可否 | この時点で戻せるか |
| 正本(SSoT) | この瞬間の正本はどこか |
| 訂正方法 | 後から間違いが見つかった時 |
| 重さ | 操作の影響の大きさ |

### 5.2 commit 点一覧表 v0.2

| commit点 | イベント | commit条件 | rollback | 正本 (SSoT) | 訂正方法 | 重さ |
|---|---|---|---|---|---|---|
| 受付登録 | 参加者の受付完了 | 氏名確認 + 会費区分確認 + 受付記録作成 | 可能 | 紙受付記録 + アプリ（ズレ時は紙/受付者確認を優先） | 削除・修正 | 軽 |
| 会費精算 | 集金完了確認 | 金額確認 + 区分確認 + 記録 | 1局目開始前は可、後は corrected | 紙受付記録 + アプリ | 訂正・精算 | 軽〜中 |
| クラス確定 | A/B クラス決定 | 受付済み + クラス選択 | 条件付き | アプリ | クラス変更モーダル | 中 |
| クラス別採番 | A/B別番号付与 | クラス確定後 | 原則不可（欠番維持） | アプリ | 欠番維持・再採番しない | 中 |
| 対局カード生成 | 組み合わせ印刷準備 | クラス別採番済み | 配布前なら可 | アプリ | 配布前なら再生成 | 中 |
| ★対局カード配布 | カードを参加者へ | 配布アクション完了 | 原則不可 | 紙 + アプリ | 全体rollback禁止・個別訂正のみ | 重 |
| 1局目開始 | 1局目開始宣言 / 対局開始 | カード配布 + 着席 + 開始案内 | 不可 | 現場 + アプリ | 個別訂正・中断判断 | 重 |
| 勝敗入力 | 勝者/敗者/結果の登録 | 対局カード回収または結果確認 + アプリ保存確認 | 直後なら条件付き(※) | アプリ | 結果訂正ログ | 軽〜中 |
| 次ラウンド生成 | 次組合せ計算 | 前ラウンド全結果 + 仮置きOK | 配布前なら可 | アプリ | 配布前なら再生成 | 中 |
| ★次ラウンドカード配布 | 次組合せ配布 | 配布アクション完了 | 原則不可 | 紙 + アプリ | 全体rollback禁止・個別訂正のみ | 重 |
| 大会終了 | 全ラウンド終了 | 最終結果確認 + 終了処理（表彰/終了宣言含む） | 不可 | アプリ + 紙台帳 | corrected/archived | 重 |
| 会員マスタ更新 | マスタ反映 | ユーザ確認 + 保存検証 | 自動不可 | 会員マスタDB | 確認付き再同期（MASTER-001方式） | 特に重 |

★太字 = 最も重要な不可逆境界（全体rollback禁止）

#### (※) 勝敗訂正の影響段階

勝敗訂正の重さは、訂正タイミングによって大きく変わる。

| タイミング | 影響範囲 | 訂正の重さ | 対応 |
|---|---|---|---|
| 入力直後（次ラウンド生成前） | 当該対局のみ | 軽 | 直接訂正 |
| 次ラウンド生成後・カード配布前 | 次ラウンド組合せに影響 | 中 | 再生成検討 |
| 次ラウンドカード配布後 | 配布物が無効化 | 重 | 個別訂正 + 現場告知 |
| 大会終了後 | 順位・表彰に影響 | 特に重 | corrected/archived 経由 |

#### 会員マスタ更新の特別な重さ

「同じ名前変更」でも、大会データ更新と会員マスタ更新は別物。

| 操作 | 影響範囲 | 重さ | 対応 |
|---|---|---|---|
| 大会データの氏名修正 | 当該大会のみ | 軽 | 即時修正可 |
| 会員マスタの氏名更新 | 長期データ・将来大会・履歴 | 特に重 | MASTER-001方式（3択モーダル + 保存検証） |

大会中にも会員マスタ更新は発生し得るが、常に published/corrected 相当の重い操作として扱う。

### 5.3 commit 点と状態遷移の対応

    [受付未]                     ← draft
      ↓ 受付登録
    [受付済]                     ← draft→validated
      ↓ クラス確定 / 採番
    [番号確定]                   ← validated
      ↓ カード生成
    [カード生成済]               ← validated
      ↓ ★カード配布
    [カード配布済]               ← validated→published
      ↓ 1局目開始
    [対局中]                     ← published
      ↓ 勝敗入力
    [結果記録]                   ← published内で更新
      ↓ 次ラウンド生成 + 配布
    [次ラウンド進行]             ← published
      ↓ 大会終了
    [終了]                       ← published→archived

各時点で訂正が発生したら corrected として記録する。

## 6. 即時整合が必要なもの

以下は同期点を待たずに即時整合させるべきデータである。
これらの不整合は「途中状態」ではなく「壊れた状態」として扱う。

### 6.1 即時整合必須リスト

| データ | 理由 |
|---|---|
| 参加者番号（クラス別） | 番号衝突は致命的。カード生成・呼出・記録すべての基盤になる |
| クラス（A/B等） | クラス間で番号体系が分かれる。クラス誤りは全体に波及する |
| 対局組み合わせ（公開後） | 配布済みカードと不一致になると現場が混乱する |
| 勝敗（確定後） | 順位計算の根幹。矛盾は許されない |
| ラウンド進行 | 「いま何回戦か」はすべての判断の前提になる |
| 同一選手の二重割当 | 物理的に1人が2局はできない |

### 6.2 即時整合チェック方針

これらは以下の方針で扱う。

- 入力時にバリデーションする
- commit 点で hard check する
- 失敗時は commit を許可しない

## 7. 一時不整合を許容するもの

以下は同期点で整合すればよく、途中状態は「警告」ではなく「正常」として扱う。

### 7.1 一時不整合許容リスト

| データ | 許容理由 | 同期点 |
|---|---|---|
| 氏名表記（大会データ） | 誤字訂正は後でできる | 受付完了時・大会終了時 |
| ふりがな | 読み違いは後修正可 | 任意 |
| 住所・電話など任意情報 | 大会進行と無関係 | 任意 |
| 支部員/一般区分 | 会費精算と連動して訂正可能 | 1局目開始前後 |
| 会員マスタとの同期 | 確認付きで明示的に行う | ユーザ操作時（MASTER-001方式） |
| 対局カード返却済みだが対戦表転記未完了 | 紙運営の自然な途中状態 | 結果集計時 |
| 勝数別の束ができ始めだが全終了ではない | 仮置き状態 | 全対局終了時 |
| 仮置き手合いが書かれているが確定発表まだ | ドラフト状態 | 一斉発表時 |
| アプリ入力済みだが紙転記未完了 | フェーズ別SSoTの自然な状態 | 同期点で整合 |

### 7.2 「途中状態」を異常表示しない原則

途中状態は以下のように扱う。

- 赤エラーにしない
- 進行ブロックしない
- ただし「同期点に到達した時点で整合が取れていない」場合は警告する

### 7.3 紙とアプリの SSoT 優先順位

フェーズ別 SSoT（A-5.0 v0.2 §4.5）を、ズレ発生時の優先順位として精密化する。

| フェーズ | 主な正本 | ズレ発生時の優先 | 訂正方向 |
|---|---|---|---|
| 受付中 | 紙受付情報・受付者確認 | 紙を優先 | アプリを修正 |
| クラス確定・採番後 | アプリ（クラス別番号） | アプリを優先 | 紙が間違っていれば書き直し |
| 対局カード配布後 | 現場に配布済みの紙カード | 紙を優先 | アプリを corrected として訂正 |
| 結果入力後 | アプリ結果 | アプリ優先。ただし対局カードと照合 | 不一致なら hard check |
| 大会終了後 | アプリ保存データ | アプリ優先。紙台帳は照合資料 | corrected / archived 経由 |

#### 優先順位の判定原則

1. 「現場に出ている物理的事実」は最も優先される
   - 配布されたカードに書かれている内容は、参加者が今手に持っている
   - これを後からアプリで書き換えると現場が混乱する

2. 「整合性が保証されたデジタルデータ」は集計フェーズで優先する
   - 採番・結果集計など、計算ロジックが介在する箇所はアプリが正確
   - 紙の転記ミスより、アプリの自動計算の方が信頼できる

3. 「ズレ発生時はどちらかを直す」のではなく「訂正イベント」として記録する
   - 単に上書きしない
   - なぜズレたか、どちらに合わせたか、いつ訂正したかを残す
   - A-5.1 初版では Minimum レベル以上で扱う

## 8. 自動同期禁止領域

以下はシステムが自動で同期してはいけない領域である。
ユーザ確認を必須とする。

### 8.1 自動同期禁止リスト

| 対象 | 理由 | 代替方法 |
|---|---|---|
| 会員マスタ氏名 | 同姓同名・表記ゆれで別人破壊リスクがある | MASTER-001方式（3択モーダル） |
| 同姓同名が絡む更新 | 自動推定は誤更新リスクが高い | 候補複数件なら更新しない |
| 支払状態 | 現金と記録の二重管理が必要 | 記録・現金・参加者の三者確認 |
| 参加履歴 | マスタ履歴は事後復旧困難 | 大会終了時に確認付き反映 |
| クラス・棋力履歴 | 長期データへの影響が大きい | 大会終了時に確認付き反映 |
| member_id 不整合時の氏名フォールバック | 別人を拾うリスクがある | 不明なら更新しない |

### 8.2 「自動同期禁止」の判定基準

以下のいずれかに該当する場合、自動同期を禁止する。

- 長期データへの影響がある
- 事後復旧が困難
- 同姓同名・表記ゆれの可能性がある
- member_id 等の信頼できる識別子で特定できない
- ユーザの意図確認が必要

## 9. ユーザ確認が必要な操作

### 9.1 確認モーダル必須リスト

| 操作 | 確認内容 |
|---|---|
| 会員マスタ更新 | 「マスタも更新する / 大会のみ修正 / キャンセル」（MASTER-001方式） |
| クラス変更（採番後） | 「採番済みです。欠番が発生します。変更しますか？」 |
| 対局カード再生成（配布前） | 「カードを再生成すると、参加者番号や組み合わせが変わる可能性があります」 |
| published 以降の組合せ変更 | 「現場に配布済みです。訂正として扱います」 |
| 勝敗の事後修正 | 「次ラウンドに影響する可能性があります。続行しますか？」 |
| 参加者削除（受付後） | 「不戦勝処理が必要です。続行しますか？」 |
| 大会終了処理 | 「未確定の対局があります。終了しますか？」 |

### 9.2 確認不要でよい操作

| 操作 | 理由 |
|---|---|
| 受付前の参加者情報修正 | draft 状態のため自由 |
| 仮置き手合いの変更 | draft 状態のため自由 |
| 入力中の氏名 typo 修正 | 即時バリデーションで十分 |

### 9.3 確認文言の基本方針

確認文言は以下を満たす。

- 何を変更するのか明示する
- どこに影響するのか明示する
- 取り消し可能かを明示する
- デフォルトは「キャンセル」側にする

## 10. エラー・保存失敗時の扱い

### 10.1 「保存できていないのに成功扱いしない」原則（MASTER-001由来）

すべての保存処理は以下の構造にする。

    save(data) を呼ぶ
      ↓
    保存後に re-read で検証
      ↓
    検証成功 → 「成功」表示
    検証失敗 → 「保存未確認」表示、リトライ提示

関数を呼んだから成功ではなく、検証できたから成功とする。

### 10.2 大会進行を止めるべきエラー

| エラー | 理由 |
|---|---|
| 参加者番号衝突 | 即時整合必須（§6） |
| 勝敗の矛盾（両者勝者等） | データ破壊 |
| 大会全体データの保存失敗 | 復旧不能リスク |
| 同一選手の二重割当 | 物理不可能 |

### 10.3 大会進行は止めず警告でよいエラー

| エラー | 対応 |
|---|---|
| 会員マスタ更新失敗 | 「大会データのみ修正」として続行(MASTER-001方式) |
| 任意情報の保存失敗 | 警告表示、後でリトライ |
| ふりがな未入力 | 警告のみ |

### 10.4 後で修正すればよいエラー

| エラー | 対応 |
|---|---|
| 氏名表記の typo | 大会終了後に corrected として修正 |
| 住所未入力 | 任意情報、後修正 |
| 棋力情報の暫定値 | 大会終了時に確定 |

### 10.5 エラー表示の3段階

| レベル | UI | 進行 |
|---|---|---|
| 致命（error） | 赤バナー、操作ブロック | 進行止める |
| 警告（warning） | 黄色バナー、操作可能 | 進行続行 |
| 通知（info） | 軽量表示 | 進行続行 |

## 11. MASTER-001 から抽出する設計原則

MASTER-001（PR #40 / a044285）から抽出する、A-5.1 全体に適用すべき設計原則。

### 11.1 「大会データ」と「会員マスタ」を分ける

短期データ（大会運営）と長期データ（会員マスタ）は同一視しない。
同期点でのみ橋渡しする。

### 11.2 「大会のみ修正」と「マスタも更新」を分ける

ユーザ操作を「現場対応」と「マスタ反映」に分離する。
両者は別の意思決定である。

### 11.3 master 更新は保存確認後に成功扱いする

verifyMasterPersisted() / verifyStatePersisted() パターンを採用する。
関数呼び出しと成功は別物である。

### 11.4 不明な場合は更新しない

候補0件 / 複数件 / 識別子不整合はすべて「更新しない」。
安全側に倒すデフォルトを徹底する。

### 11.5 自動推定しすぎない

yomi 一致での自動推定はしない。
推定が便利でも、誤更新リスクの方が大きい。

### 11.6 大会運営は止めすぎない

会員マスタ更新ができなくても、大会データの修正は進める。
マスタ不確実性が大会進行を止めるべきではない。

### 11.7 保存成功を嘘つかない

「内部実装で握りつぶし」を想定して、外側で検証する。
これは Codex Must Fix の本質である。

### 11.8 欠番は許容する（独立原則）

クラス別採番後に削除・取消が発生しても、番号を詰め直さない。

#### 理由

- 既に対局カード・呼び出し・記録が、その番号前提で動いている
- 再採番は現場の混乱を引き起こす
- 誰の番号が変わったかを全員に告知する必要が出る
- 欠番は単なる「見た目の問題」だが、再採番は「現場破壊」である
- 「再採番の混乱 > 欠番の見た目の問題」と考える

#### 実装上の含意

- 採番ロジックは「次の番号を採る」だけにする
- 削除発生時に番号を詰めない
- 表示上「欠番です」と明示するのは可

これは独立した設計原則として A-5.1 全体に適用する。
