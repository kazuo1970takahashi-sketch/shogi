## 2. Minor 推奨5件

### M1. parseTournamentTextInput() のエラー仕様を明確化

**指摘**: 仕様書 §3.1.4 で「失敗時はエラーメッセージを集約して throw」と書かれているが、返り値は tournaments と errors を返す形になっており、throw と return の責務が混在している。

**推奨仕様**:
- parseTournamentTextInput(text) は **原則 throw しない**
- 以下の形で返す：
  ```javascript
  {
    tournaments: [...],
    errors: [...]
  }
  ```

**挙動**:
- 空文字 / 空白のみ: `tournaments: [], errors: []`
- 全件パース失敗: `tournaments: [], errors: [...]`
- 一部成功: `tournaments: [成功分], errors: [失敗分]`
- 想定外の内部例外のみ throw

**イベントハンドラ側**:
- `tournaments.length === 0 && errors.length === 0`: 「大会データを貼り付けてください」
- `tournaments.length === 0 && errors.length > 0`: 「貼り付けたデータの形式が正しくありません」
- `tournaments.length > 0 && errors.length > 0`: 成功分は統合し、「N件中M件を読み込みました。X件は失敗しました」と表示

### M2. 複数大会の空行区切り仕様を明確化

**指摘**: 分割条件を「JSONブロック間の空行」と明確化し、末尾空行・複数空行・CRLF/LF/CR をテストに入れる。

**推奨仕様**:
- CRLF / LF / CR を **正規化してから分割**
- BOM は **先頭から除去**
- 分割後 trim() して **空ブロックは捨てる**
- **連続空行は1つの区切りとして扱う**

### M3. 既存ファイル保有者向けの代替案内

**指摘**: ファイル選択 UI を完全撤廃する場合、既に .json ファイルとして保存済みの人への説明をUIまたはマニュアルに1文入れる。

**推奨**: マイグレモーダルの補足に以下のような1文を追加：
> 「ファイルとして保存してある場合は、ファイルを開いて中身をコピーし、ここに貼り付けてください。」

### M4. 「JSON」撤廃は UI 限定であることの再明記

**指摘**: UI文言からの「JSON」撤廃と、内部コメント・コード識別子・ファイル拡張子 `.json` を変更しない方針を、実装依頼で再明記する。

**推奨**: 実装依頼の禁止事項に明記：
- UI文言の「JSON」撤廃を、内部コードの JSON.parse / JSON.stringify まで広げない
- ファイル拡張子 .json を変更しない
- 内部識別子（saveBtn, json 変数名等）を大規模リネームしない

### M5. 「読み込み」と「マスタ統合」の違いを画面で説明

**指摘**: 既存の「読み込み」機能と「マスタ統合」機能の違いを、画面上で短く説明する。

**推奨UI文言**:
- マイグレモーダル内: 「過去大会の参加者を支部マスタに取り込みます。大会そのものを復元する機能ではありません。」
- 読み込み画面側: 「大会データ全体を復元します。」

## 3. 主要設計判断（v6.5 §1.3）への評価

### 3.1 マイグレ入力方式（テキスト貼り付け vs ファイル選択 vs 両対応）

**判定**: **テキスト貼り付け採用で妥当**

| 選択肢 | 評価 |
|---|---|
| **テキスト貼り付け（採用）** | ✅ 保存導線と整合、スマホで分かりやすい、既存読み込み UI と思想統一 |
| ファイル選択維持 | ❌ 構造矛盾が残る、スマホで認知負荷高 |
| 両対応 | ❌ A-2.5 の小規模性に反する、迷いが出る、A-3 F8 と責務混在 |

**結論**: A-2.5 ではテキスト貼り付け一本化が最適。ただし既存ファイル保有者向けに M3 の案内を追加。

### 3.2 「JSON」用語の扱い（完全撤廃 vs 部分残置 vs 完全維持）

**判定**: **UI から完全撤廃が妥当**

| 選択肢 | 評価 |
|---|---|
| **UIから完全撤廃（採用）** | ✅ 認知負荷削減、説明しやすい |
| 部分残置 | ❌ 中途半端、混乱を生む |
| 完全維持 | ❌ 運営者に不親切、実機テストの違和感を解消できない |

**結論**: UI からは完全撤廃。内部コード、JSON.parse / JSON.stringify、拡張子 .json は維持。

### 3.3 保存ボタン方式（現状維持 vs ファイル化 vs 撤廃）

**判定**: **現状維持が妥当**

| 選択肢 | 評価 |
|---|---|
| **現状維持（採用）** | ✅ 既存資産、A-2.5 の目的に合致 |
| ファイル化 | ❌ スマホ運用の負荷が上がる、現行導線を壊す |
| 撤廃 | ❌ 危険、明示バックアップ導線が消える |

**結論**: 保存ボタン方式は現状維持。文言だけ「大会データをコピー」に変更が最小で安全。

### 3.4 複数大会の区切り（空行 vs 専用区切り文字 vs 専用フィールド）

**判定**: **空行区切りで妥当**

| 選択肢 | 評価 |
|---|---|
| **空行（採用）** | ✅ 簡単、コピペ運用に合う、説明しやすい |
| 専用区切り文字 | ❌ 誤操作リスク、非技術者向けでない |
| 専用フィールド | ❌ A-2.5 には過剰 |

**結論**: 空行区切り。ただし説明文とテストを補強（M2）。
