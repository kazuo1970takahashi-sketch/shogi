## 3. レビュー観点

### ★★★ 必須レビュー観点

#### A. データモデルの後方互換性

A-1/A-2 で構築済みのマスタ（schema_version: 1、`deleted: false` が大半）を A-3 が壊さないか：

- v7 §2.1：schema_version 変更なし、deleted/deleted_at は v5 で予約済み
- v7 §2.2：A-3 で削除（tombstone）した member を A-1/A-2 が読んでも壊れないか
- 既存の `members.filter(m => !m.deleted)` ロジックが効くか

**確認事項**：
- マスタ JSON スキーマレベルでの後方互換性
- A-3 でエクスポートした JSON を A-2 端末でインポートしても破綻しないか
- tombstone を持つマスタが A-2 で読み込まれた時の挙動

#### B. F7 編集スコープの妥当性

v7 §3.1.2：name + yomi のみ編集可能、参加履歴（last_attended、attendance_count、first_attended、tournament_ids）は読み取り専用。

**確認事項**：
- 参加履歴を編集不可にする判断は妥当か（誤入力修正の柔軟性 vs 履歴の信頼性）
- 「同名重複の警告のみ（強制統合しない）」運用方針は妥当か
- 編集後の同名検出時の運用フロー（warn のみで OK か、それともブロックすべきか）

#### C. F7 削除の tombstone 方式

v7 §3.1.3：物理削除しない、`deleted: true` のみ立てる。A-3 では復元 UI なし。

**確認事項**：
- 物理削除を提供しない判断の妥当性
- 復元 UI なしで運用に支障ないか（誤削除時の救済路）
- 「LocalStorage 直接編集で復元可能」を救済路として想定する是非

#### D. F8 戦略 Z（上書き + マージ）の妥当性

v7 §3.2.3：ラジオボタンで「上書き」「マージ」を選択。

**確認事項**：
- 戦略 Z（両方サポート）が UI として複雑になりすぎないか
- 「上書き」を誤選択した場合のデータ消失リスク（確認ダイアログで防げるか）
- 戦略 X（上書きのみ）/ 戦略 Y（マージのみ）と比べて穴はないか
- インポート時の不正データ検証（v7 §3.2.4）の妥当性

#### E. クイックフィルタ3種 + 排他選択

v7 §3.3：「最終参加日 直近」「3ヶ月以内」「常連 (5回以上)」の3種、排他選択 + 再タップで解除。

**確認事項**：
- 3種類のフィルタの選定（過不足ないか）
- 排他選択（vs 複数選択 AND）の妥当性
- 「常連」の閾値5回ハードコードは妥当か
- 50音タブ + 検索 + フィルタの3条件 AND の妥当性

### ★★ 重要レビュー観点

#### F. YAGNI / 過剰設計

A-3 のスコープに対して仕様が大きすぎないか：

- F8 戦略 Z（上書き + マージ）は片方だけで済まないか
- クイックフィルタ3種を一気に作る必要があるか（最終参加日 直近だけで十分？）
- F7 編集モーダルの必要項目（参加履歴の表示は読み取り専用でも見せる必要があるか）

#### G. 既存機能への影響

A-1/A-2 で完成した機能（漢字検索、ふりがな検索、50音タブ、ワンクリック呼び出し、マイグレ）が A-3 で壊れないか：

- マスタタブの簡易一覧に「操作」列追加が既存表示を壊さないか
- 過去参加者パネルにクイックフィルタ追加が UI を破壊しないか
- 既存テスト 306 件すべて緑のまま維持できるか

#### H. テスト計画の十分性

v7 §5 のテスト計画：

**確認事項**：
- §5.2 の単体テスト 7カテゴリで網羅性は十分か
- §5.3 の性質テスト（id 維持、tombstone 物理削除なし、エクスポート→インポートで完全復元）が妥当か
- 不正データ系のテスト（破損 JSON、schema_version 違い）が十分か

#### I. 実装順（Stage 1〜8）の妥当性

v7 §6 のステージ分け：

**確認事項**：
- 順序の妥当性（F7-a → F7-b → F8 上書き → F8 マージ → クイックフィルタの順）
- Stage 数が増えている（A-2 の 6 → A-3 の 8）が肥大化リスクは管理できるか
- 各 Stage の独立性（途中で運用に出せる粒度か）

### ★ 補助レビュー観点
