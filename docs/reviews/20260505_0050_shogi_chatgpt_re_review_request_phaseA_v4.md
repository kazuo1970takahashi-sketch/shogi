## 3. 再レビューの主要観点

### 観点1：Must Fix 3点の反映十分性（必須判定）

各 Must Fix について、v4 で反映した内容が **適切・十分** か判定してください。

#### MF1: A-1 / A-2 の矛盾解消

**v4 の対応**：F11「最小マスタタブ」を A-1 必須機能として新規追加（推奨案1）。

確認項目：
- 4.2.1 節「最小マスタタブ（F11、A-1 必須）」の仕様は適切か
- 9章「Phase A-1 再定義」で「最小マスタタブ」が含まれているか
- F9 マイグレーション起動経路の矛盾が解消されているか
- A-1 で含めない機能（編集・削除・エクスポート/インポート・検索フィルタ）の線引きが明確か

判定：**OK / Minor / Must Fix が残っている**

#### MF2: member_id を 12文字以上に

**v4 の対応**：`m_` + `crypto.randomUUID().slice(0, 12)` + 衝突再生成ロジック。

確認項目：
- 3.1 のフィールド説明、3.3 の生成ルールが整合しているか
- 衝突検出・再生成のロジック（最大3回試行）が記述されているか
- 生成済み ID は変更しない（shogi-coach 連携の安定性）と明記されているか

判定：**OK / Minor / Must Fix が残っている**

#### MF3: normalizeBranchMaster(raw) 仕様の明記

**v4 の対応**：3.5 節を新設。破損パターン全網羅の正規化仕様。

確認項目：
- schema_version 不在・未知への対応が明記されているか
- members 配列・各フィールドの補正ルールが網羅的か
- 重複 tournament_id 除去、attendance_count 再計算が含まれているか
- 「壊れたマスタを読んでも既存大会運営を止めない」哲学が貫かれているか
- 7.4 マスタ破損時テストと整合しているか

判定：**OK / Minor / Must Fix が残っている**

## 4. 期待するアウトプット

### 形式

1. **Must Fix 3点それぞれの判定**（OK / Minor / Must Fix が残っている）
2. **Minor 11件の判定サマリ**（OK 中心、残課題あれば明記）
3. **v4 新規導入箇所の判定**（新規リスクの有無）
4. **総合判定**：
   - **Go**：そのまま実装に進める
   - **Conditional Go**：軽微な修正のみで実装可
   - **No-Go（追加 Must Fix あり）**：v5 が必要
5. **等級判定**（v3 が B+ だったので、v4 が向上したか同等か）

### 特に避けたい落とし穴

- 反映が形式的だけで実質を伴っていない箇所を見過ごす
- 新たに導入した仕様（最小マスタタブ、normalizeBranchMaster 等）に潜む問題を見過ごす
- 仕様書のサイズ膨張（770行）に伴う一貫性の崩れを見過ごす
