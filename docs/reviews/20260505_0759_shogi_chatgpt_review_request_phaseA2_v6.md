## 3. レビュー観点

### ★★★ 必須レビュー観点

#### A. データモデルの後方互換性

A-1 で構築済みのマスタ（schema_version: 1、yomi 空が大半）を A-2 が壊さないか：

- v6 §2.1：schema_version 変更なし、yomi フィールドは v5 で予約済み
- v6 §2.2：A-1 で yomi 空、A-2 で yomi 入力開始のフロー
- A-2 で yomi を追加したマスタを A-1 が読んでも壊れないか

**確認事項**：
- マスタ JSON スキーマレベルでの後方互換性が保たれているか
- 既存 member の yomi 空のまま運用しても A-2 機能（検索・タブ）が破綻しないか

#### B. ふりがな入力フロー（方式 Y、パターン B）の妥当性

v6 §3.1 のフロー：
```
ユーザーが過去参加者パネルでチェック → 「追加」ボタン押下
  ↓
追加対象 N 名のうち yomi 空の人だけ抽出
  ↓
yomi 空 0名 → 即座に追加完了
yomi 空 1名以上 → ふりがな一括入力ダイアログ表示
```

**確認事項**：
- 方式 X（手入力のみ）/ Z（フォームに yomi 欄追加）と比べて穴はないか
- 「キャンセル」を押した時の挙動：v6 では「何もしない（追加しない）」だが、これでよいか？「yomi 空でもとりあえず追加」のほうが UX 良くないか
- 既に yomi が登録されている member は「ダイアログに含めない」設計（v6 §3.1）でよいか
- ダイアログでの空欄入力（スキップ）が許容されているが、これが「全員空欄でスキップ」の運用に陥らないか

#### C. 50音タブ仕様の妥当性

v6 §3.3 の12タブ構成（全/あ/か/さ/た/な/は/ま/や/ら/わ/他）：

**確認事項**：
- yomi 空の人が「他」タブに集約される設計（パターン あ）の妥当性
  - 「探したい人が消える」事故を防ぐ意図だが、運用上の違和感はないか
- 「他」タブに yomi 空 + 外国名 + 数字始まりが混在する設計でよいか
  - 「未登録の人」と「外国名」を区別したいケースは出ないか
- 各行タブと検索ボックスの AND 条件（v6 §3.3）の妥当性
- 「全」タブをデフォルトにする設計でよいか

#### D. normalizeYomi / getYomiInitialRow の関数仕様

v6 §3.4-3.5 のユーティリティ関数：

**確認事項**：
- カタカナ → ひらがな変換の境界（ヴ、ヵ、ヶ など）
- 長音符（ー）保持の妥当性
- 小書き文字（ぁ、ょ、っ）の50音行判定（getYomiInitialRow で「小書き ぁ → あ行」「小書き ょ → や行」「小書き っ → た行」）
- 濁音・半濁音（が→か行、ぱ→は行）の行判定
- yomi の空白（途中含む）を全部削除する設計でよいか

#### E. F5b 検索の OR ヒット動作

v6 §3.2：「漢字一致 OR ふりがな一致」でヒット。

**確認事項**：
- 両方一致する member（漢字とふりがな両方ヒット）が二重表示されないか
- 「sa」のようなローマ字入力をヒットさせない設計でよいか
- 検索文字列の `normalizeYomi` 適用は妥当か

### ★★ 重要レビュー観点

#### F. YAGNI / 過剰設計

A-2 のスコープに対して仕様が大きすぎないか：

- 「Codex Minor #1: 破損マスタ警告表示」（v6 §3.6）は A-3 でもよいのでは？
- 50音タブの12分割は細かすぎないか（5分割「あいうえ / かきくけこ ... 」のような統合案はないか）
- ふりがな入力ダイアログを A-2 の最小実装で省略する案はあるか

#### G. 既存機能への影響

A-1 で完成した機能（漢字検索、ワンクリック呼び出し、マイグレ）が A-2 で壊れないか：

- 検索ボックスのプレースホルダ変更（「漢字部分一致」→「漢字・ふりがな」）の互換性
- ワンクリック呼び出しの「追加」ボタン挙動の変更（ダイアログ追加）が既存の使い勝手を損なわないか
- 既存テスト 161 件すべて緑のまま維持できるか

#### H. テスト計画の十分性

**確認事項**：

- §5.3 の性質テスト「N 名の50音行合計が N と一致」だけで十分か
- §5.4 の非破壊テストの範囲

#### I. 実装順（Stage 1〜7）の妥当性

v6 §6 のステージ分け：

**確認事項**：
- 順序の妥当性（ユーティリティ → F5b → F6 → F4 → Codex Minor の順）
- 各 Stage の独立性（途中で運用に出せる粒度か）
- 1 Stage = 1 commit の粒度が適切か
