### 1.2 Phase A-2.5 の発端（実機テスト由来）

A-3 v7 設計を完了し ChatGPT メタレビュー依頼を準備中、髙橋さんが実機テスト中に **構造的矛盾** を発見：

1. **「JSONバックアップ保存」ボタン文言と実装の不一致への違和感**
2. **「JSON」という用語が運営者（将棋愛好家、技術者ではない）に伝わらない**
3. **マスタタブの「📥 過去大会を支部マスタに統合」が JSON ファイル選択を要求するが、通常運用ではアプリ上で JSON ファイルを生成する手段がない**（保存はクリップボードコピー優先のため）

実装の正確な状態を Claude Code に読み取らせて把握：

```javascript
// saveData() 実装（2111-2125行目）
function saveData() {
  syncBranchMasterOnSave();
  var json = JSON.stringify(state, null, 2);
  
  // クリップボード優先（HTTPS環境）
  if (navigator.clipboard && navigator.clipboard.writeText && window.isSecureContext) {
    navigator.clipboard.writeText(json).then(...);
    return;
  }
  
  // フォールバック：ファイルダウンロード
  saveDataAsFile(json);
}
```

```javascript
// マイグレモーダル UI（1057行目）
'<input type="file" id="mig-files" accept=".json" multiple>'
```

つまり **保存はクリップボード優先なのに、マイグレだけはファイル選択強制** という非対称性。

### 1.3 v6.5 の方針

| 方針 | 内容 |
|---|---|
| **マイグレ UI を貼り付け方式に変更** | ファイル選択を撤廃、テキストエリアに統一 |
| **「JSON」用語を運営者向けに置換** | UI 文言から「JSON」を撤廃 |
| **保存ボタンは現状維持** | クリップボード優先 + ファイルフォールバックは維持 |
| **保存→マイグレの動線を一貫** | クリップボードコピー → 貼り付けマイグレ |

## 3. レビュー観点

### ★★★ 必須レビュー観点

#### A. 構造矛盾解消の妥当性

v6.5 §3.1：マイグレ機能を **テキスト貼り付け方式** に変更、ファイル選択 UI を撤廃。

**確認事項**：
- 保存（クリップボード優先）→ マイグレ（貼り付け）の動線が一貫しているか
- ファイル選択 UI 撤廃で運用に支障が出るケース（過去JSONファイル群を持っている運営者）
- 既存の `load-paste-area`（大会データ全体の読み込み）と整合的か

#### B. 用語整理のスコープ

v6.5 §3.2-3.3：UI 文言から「JSON」を完全撤廃、内部識別子は維持。

**確認事項**：
- UI 文言と内部識別子（id/関数名/JS の JSON.stringify）を分離する判断の妥当性
- ファイル名 `shogi_YYYYMMDD_HHMM.json` の拡張子を変更しない判断
- 「大会データをコピー」「大会データを貼り付け」「大会データを取り込んで」等の置換語が運営者にとって自然か

#### C. 後方互換性

**確認事項**：
- 既存マスタ・既存大会データ（schema_version: 1）への影響なし
- A-1/A-2 の機能（306アサーション）が緑のまま維持
- 既存の `mergeTournamentParticipantsIntoMaster()` が入力経路（テキスト/ファイル）に依存しない

#### D. UX への影響

**確認事項**：
- 運営者が「コピー → 貼り付け」フローに違和感なく移行できるか
- alert メッセージの追加情報「次回大会前に『マスタ』タブの統合機能で取り込めます」が誘導として機能するか
- 複数大会を **空行区切り** で貼り付ける UI の説明が分かりやすいか
- 既存の `load-paste-area`（大会データ全体読み込み）と新マイグレ UI（マスタ統合）が混在することによる混乱

#### E. parseTournamentTextInput() の堅牢性

v6.5 §3.1.4：新規ユーティリティ関数の仕様。

**確認事項**：
- 空行検出ロジック（連続改行で大会区切り）の妥当性
- BOM 付き UTF-8 対応（運営者が Word からコピペする可能性）
- 改行コード差異（CRLF/LF/CR）対応
- 一部失敗時のエラーレポート（成功した分は処理する）
- 不正データ（schema_version 不一致、構造違反）の検出
