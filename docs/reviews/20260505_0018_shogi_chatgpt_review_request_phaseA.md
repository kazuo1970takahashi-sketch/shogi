## 1. レビューを依頼する背景

shogi_v4 アプリに、月例大会の **受付効率化機能** を追加する。

### 提案機能（要約）
- **支部マスタ**：localStorage に「過去参加者の氏名・参加履歴」を蓄積する DB 層を新設
- **ワンクリック呼び出し**：新大会セットアップ時、過去参加者をチェックボックスで一括追加
- **検索フィルタ**：マスタが累積する将来を見越し、漢字部分一致検索を A-1 に同梱
- **A-2（後フェーズ）**：ふりがな・50音タブ・マスタ管理画面・エクスポート/インポート

### なぜ今このレビューか
- データモデルの新設は、ChatGPT が 2026-05-01 ロードマップ評価で
  「**大会履歴を何年分も管理したい場合はデータモデルの見直しが必要**」と
  指摘した領域に該当する
- 単一HTML＋localStorage アーキテクチャの延長で本当にいけるかを確認したい
- 実装に着手する前に、設計レベルで問題を潰しておきたい

### 過去レビューの前回リファレンス
2026-05-01 のロードマップ評価（当時 "Codex 評価" として記録、実体は ChatGPT GitHub 連携によるレビュー）では、以下のメッセージをいただいた：
- 「次の一手は機能を増やすより**運用の信頼性を上げる**」
- Phase 0（安定運用の土台）→ Phase 1（実運用フィードバック収集）の順で進めるべき

本仕様は **Phase 1（運用フィードバック収集）に直結する受付効率化** という位置づけ。
今回の Phase A 内部分割（A-1 → 運用試験 → A-2）も、この哲学を反映している。

## 3. レビュー観点（優先度順）

### ★★★ 必ず判定してほしいもの

#### A. データモデルの妥当性
- `schema_version` 設計：前方互換／後方互換は十分か
- tombstone方式：`deleted: true` で論理削除する方針に穴はないか
- ID付与ルール：`m_` + UUID先頭6文字での衝突確率は本当に無視できるか
- 中量版データモデル（参加履歴を `tournament_ids[]` で持つ）の妥当性
- マスタ更新ロジック（曖昧マッチ → 対話確認）に落とし穴はないか

#### B. アーキテクチャ転換ポイントとの距離
- 単一HTML＋localStorage の延長で **本当にいけるか**
- このマスタ機能を入れることで、サーバー化が近づいていないか
- 5年運用で localStorage 容量限界に当たる可能性は
- 2026-05-01 評価の「アーキテクチャ転換ポイント」のどこに今いるか

#### C. HANDOFF.md との整合
- 既存の設計思想（単一HTML、localStorage 完結、PoC 段階の常識的扱い）に反していないか
- 既存の編集ルール・運用ルールに矛盾はないか
- リファクタ履歴で禁忌とされた領域に踏み込んでいないか

### ★★ 重要だが穴があれば気付きたいもの

#### D. YAGNI / 過剰設計の指摘
- 不要な機能はないか（特に A-2 機能群）
- tombstone より物理削除のほうが単純で十分ではないか
- `tournament_ids[]` の保持は本当に必要か（軽量版で十分ではないか）

#### E. A-1 / A-2 分割の妥当性
- A-1 のスコープは小さすぎ／大きすぎではないか
- 漢字検索（F5a）を A-1 に含めた判断は妥当か
- A-1 単体で運用試験できる完結性があるか
- A-2 の機能を A-1 に前倒しすべきものはないか

#### F. テスト計画の十分性
- 性質テストでカバーすべきパターンに漏れはないか
- マイグレーションのエッジケース（同名異人、ふりがな揺れ等）への備え
- 既存テスト（test/）との接続性

### ★ 余裕があれば

#### G. その他見落とし観点
- 仕様書に書かれていないが書いておくべきこと
- 髙橋さんの運用想定で抜けていそうな観点
- 将来の Phase B（キオスクモード）との接続で問題になりそうなこと
- 個人情報の取り扱い（支部内利用前提）の妥当性

## 4. 期待するアウトプット

### 形式
- 上記レビュー観点 A〜G について、それぞれ判定とコメント
- A〜C（★★★）は **必ず** 言及してほしい
- D〜F（★★）は穴があれば指摘
- 全体評価（A / A- / B+ / B / C などの等級判定）

### 判定の解像度
- **OK**：そのまま実装に進んでよい
- **Minor**：軽微な改善提案あり（実装中に対応可）
- **Must Fix**：実装前に仕様書を修正すべき
- **Architecture Concern**：方針レベルで再検討が必要

### 特に避けたい落とし穴
- 「データモデルの見直しゾーン」に踏み込んでいるのに見過ごす
- HANDOFF.md と矛盾する設計を見過ごす
- A-1 のスコープが膨らみすぎていることを見過ごす
- 単一HTML アーキテクチャの限界が近づいているのに気付かない

## 5. 質問にしておきたいこと（任意回答）

判定とは別に、ChatGPT の見解を聞きたい点：

1. **localStorage の限界**：何名 × 何年分が現実的な上限か。実測ベースで助言できるか
2. **ふりがな自動取得**：IME API の信頼性は実用レベルか（A-2 でユーザ手入力 vs 自動補完）
3. **支部展開**：将来、他の将棋支部にも配布する場合、現設計の流用度はどの程度か
4. **shogi-coach 連携**：別プロジェクト shogi-coach（KIF 解析）との将来連携を視野に入れた時、本マスタ設計に追加すべき布石はあるか
